# Roadmap 10x — Pasture

> Generado el 2026-07-05 a partir de un análisis multi-agente (43 agentes): 3 lectores del código y del mercado, 5 perspectivas de ideación independientes (25 ideas brutas), dedup semántico a 9 candidatas, panel de 3 jueces por candidata (lentes: impacto de usuario, viabilidad/identidad, diferenciación) y especificación anclada en el código para las 7 finalistas.

## Tesis

Pasture hoy es una biblioteca estática con un canal de salida excelente. Las 7 capacidades de este roadmap la convierten en otra cosa: **la capa de contexto local, curada por humano y auditable, para todos los agentes del usuario**. Tres movimientos: (1) explotar la capa MCP ya construida para que el vault sea consumible desde cualquier cliente sin copy-paste; (2) cerrar el loop de memoria (los agentes proponen, el humano aprueba, el contexto declara su frescura); (3) hacer que el vault escale (relevancia on-device, observabilidad de uso). Todo dentro de la identidad: nativo macOS, cero dependencias, privacidad primero.

## Fase 1 — v1.6 «El hub MCP completo»

**Por qué esta fase:** Máximo valor con mínimo riesgo: multiplicar los activos ya construidos (TemplateEngine + capa MCP + presets). Pasture pasa de "app que abres para pegar contexto" a infraestructura permanente de todos tus agentes.

### pasture-mcp v1.6: templates como MCP prompts y archivos de la vault como MCP resources

**Score del panel:** 7.7/10 · **Esfuerzo:** 1.5 semanas

**Qué es:** Ampliar pasture-mcp con las otras dos primitivas del protocolo: resources/list + resources/read (cada .md de la vault, reutilizando enumerateVaultFiles y los caps de MCPLimits) y prompts/list + prompts/get (cada template se expone como prompt parametrizado: extractVariables() genera los argumentos tipados, el render de una pasada produce el mensaje, con el guard de secretos post-sustitución de ADR-QW-002). En Claude Code, tus templates aparecen como slash-commands con argumentos; en Claude Desktop, tus notas son adjuntables como resources nativos.

**Por qué es 10x:** La relación valor/coste más alta de la lista: multiplica dos activos ya construidos (TemplateEngine completo y capa MCP endurecida) cruzándolos. Los templates dejan de ser una feature de la GUI para ser una biblioteca de prompts reutilizable desde cualquier cliente MCP — el hueco de los prompt managers (PromptLayer et al.) pero local, privado y sin cloud. Ningún gestor de contexto expone hoy las tres primitivas MCP desde una vault personal. Riesgo técnico mínimo: read-only, capabilities ya se negocian en initialize, caps SEC-M3/M5 heredados, todo testeable contra MCPDispatcher.handle(line:) sin spawn de proceso.

**Veredictos del panel (impacto / viabilidad / diferenciación):**

- La mitad de prompts es un multiplicador genuino de uso diario: convierte los templates (hoy encerrados en la GUI) en slash-commands con argumentos justo donde el usuario de Pasture ya trabaja (Claude Code), y eso se usaría muchas veces al día, no una. La mitad de resources es bastante más marginal: los tools existentes (read_file, search, feed_context) ya cubren el acceso al contenido, y su valor real se reduce a la UX de adjuntar/@-mencionar en los clientes que lo soporten — un nice-to-have. El impacto neto depende de cuánto use templates el usuario tipo; para quien los usa, es de las pocas features de la lista que cambia el flujo diario en vez de añadir un botón.
- Verificado en el código: la propuesta recombina piezas existentes y públicas de PastureKit — MCPDispatcher.handle(line:) es un switch extensible, TemplateEngine.extractVariables/render ya son API pública, y enumerateVaultFiles + MCPLimits + SecretScanner cubren resources sin código nuevo de I/O. Respeta todos los invariantes: sigue siendo estrictamente read-only (prompts/get renderiza en memoria, no escribe), cero dependencias, y el guard de secretos post-sustitución (ADR-QW-002) ya existe tal cual. El esfuerzo de 1.5 semanas es realista e incluso conservador dado el patrón de tests contra el dispatcher sin spawn de proceso; el único trabajo estructural real es ampliar InitializeResult.Capabilities (hoy solo declara tools) y decidir qué archivos cuentan como template (hasTemplateVars ya lo resuelve). El riesgo principal no es técnico sino de adopción: soporte desigual de resources/prompts entre clientes MCP, correctamente identificado en la propia candidata.
- La primitiva aislada es copiable (un MCP server genérico que exponga resources/prompts es un fin de semana, y Claude Code ya da slash-commands gratis vía .claude/commands/*.md), pero lo que un competidor no replica rápido es el cruce completo: TemplateEngine con argumentos tipados (#if/#each, defaults), SecretScanner post-sustitución, capa MCP endurecida con caps de seguridad y una GUI local-first que gestiona la misma vault — eso sí explota la posición única de Pasture. El solapamiento real es con los slash-commands nativos de Claude Code para el usuario mono-cliente, que ya cubren el 60% del caso de uso sin Pasture; la diferenciación depende de que el usuario valore multi-cliente, parametrización rica y el guard de secretos. Refuerza el foso existente más que crear uno nuevo: buena inversión, diferenciación moderada.

#### User stories

- Como usuario de Claude Code con Pasture registrado, quiero que cada template de mi vault (~/.pasture/) aparezca como prompt MCP con argumentos tipados, para invocar mis templates como slash-commands sin abrir la GUI de Pasture.
- Como usuario de Claude Desktop, quiero que cada .md de mi vault se exponga como resource MCP nativo, para adjuntar notas a una conversación con @-mención en lugar de pedirle al modelo que llame a read_file.
- Como autor de templates, quiero que las variables extraídas por TemplateEngine.extractVariables() se conviertan en argumentos del prompt (con su default y su kind .scalar/.list documentados), para que el cliente MCP me pida exactamente los valores que el template necesita.
- Como usuario consciente de la privacidad, quiero que el resultado de prompts/get pase por SecretScanner sobre el contenido YA renderizado (post-sustitución, ADR-QW-002), para enterarme si un valor de argumento o el propio template inyectan una credencial antes de que llegue al modelo.
- Como mantenedor del proyecto, quiero que las nuevas primitivas se despachen por MCPDispatcher.handle(line:) igual que las existentes, para testearlas sin spawn de proceso y heredar SEC-M12 (una request inválida nunca tumba la conexión).
- Como usuario con vault grande, quiero que resources/read y prompts/get respeten los caps de MCPLimits (SEC-M3/SEC-M5), para que un fichero de 100 MB no reviente el cliente MCP ni la RAM del servidor.

#### Criterios de aceptación

1. DADO un servidor recién arrancado, CUANDO el cliente envía initialize, ENTONCES la respuesta incluye capabilities.tools, capabilities.resources y capabilities.prompts, los tres presentes aunque vacíos ({}) — verificable con un test en MCPProtocolTests que decodifica la línea y comprueba las tres claves (mismo gotcha que hoy documenta InitializeResult para tools).
2. DADO un vault con notas.md en la raíz y proyecto/spec.md en una colección, CUANDO el cliente envía resources/list, ENTONCES la respuesta contiene exactamente 2 resources con uri pasture:///notas.md y pasture:///proyecto/spec.md, name = ruta relativa, mimeType text/markdown — verificable con test sobre MCPDispatcher.handle(line:) contra un vault temporal.
3. DADO un symlink dentro del vault que apunta a /etc/passwd, CUANDO el cliente envía resources/list y resources/read, ENTONCES el symlink no aparece en la lista (FileLibrary ya lo filtra) y resources/read con su uri devuelve isError/-32602 con mensaje 'ruta fuera del vault', sin realizar I/O — verificable con test que crea el symlink y comprueba ambas respuestas (SEC-M2, dos capas de MCPPathResolver).
4. DADO un uri pasture:///../../.ssh/id_rsa o un uri con esquema file://, CUANDO el cliente envía resources/read, ENTONCES la respuesta es error de tool sin leer el fichero — verificable con test parametrizado de traversal (SEC-M1).
5. DADO un fichero de más de 25 MB en disco, CUANDO el cliente envía resources/read con su uri, ENTONCES la respuesta es error ('fichero demasiado grande') SIN materializar el contenido en RAM (comprobación previa con fileSizeOnDisk, patrón idéntico a read_file) — verificable con test que crea el fichero con truncate y mide que la respuesta llega (SEC-M5).
6. DADO un vault con un fichero con {{VAR}} y otro sin variables, CUANDO el cliente envía prompts/list, ENTONCES solo el primero aparece como prompt (filtro = TemplateEngine.hasVariables(in:) || hasBlocks(in:), misma semántica que MDFile.hasTemplateVars) — verificable con test contra vault temporal.
7. DADO un template con {{NOMBRE}}, {{TONO=formal}} y {{#each ITEMS}}…{{/each}}, CUANDO el cliente envía prompts/list, ENTONCES el prompt declara 3 argumentos: NOMBRE required=true, TONO required=false con el default 'formal' citado en su description, ITEMS required=true con description que documenta la convención de valores separados por comas (VariableKind.list → listItems) — verificable con test que decodifica el catálogo.
8. DADO prompts/get con name válido y arguments {"NOMBRE":"Ana","TONO":""} (TONO omitido o vacío), CUANDO se procesa, ENTONCES messages contiene un único mensaje role=user cuyo texto es el render de una sola pasada de TemplateEngine.render (Ana sustituido, TONO resuelto a 'formal'), y un valor de argumento que contenga {{OTRA}} aparece literal en la salida, NUNCA re-interpretado como sintaxis de template — verificable con test de inyección (single-pass, invariante existente del motor).
9. DADO prompts/get donde el valor de un argumento es 'sk-ant-api03-…' (clave sintética construida por concatenación, como el fixture ASIA), CUANDO se renderiza, ENTONCES el contenido se entrega íntegro y el campo description del GetPromptResult lleva el resumen ENMASCARADO de SecretScanner.summaryLines() (familia + fichero, jamás el valor completo) — verificable con test que comprueba que la respuesta no contiene la clave enmascarada completa pero sí el aviso (ADR-QW-002 + SEC-M8/D4: informativo, no gate).
10. DADO prompts/get con un name inexistente o con un argumento required ausente, CUANDO se procesa, ENTONCES la respuesta es JSON-RPC error -32602 (invalidParams) según spec MCP para prompts — canal de error de PROTOCOLO, verificable con test que distingue este caso del isError de tools (SEC-M12).
11. DADO una sesión real, CUANDO se ejecuta la secuencia initialize → resources/list → resources/read → prompts/list → prompts/get línea a línea contra MCPDispatcher, ENTONCES cada respuesta es una única línea JSON válida con claves ordenadas y sin '/' escapado como '\/' en los uri (ADR-MCP-006, .withoutEscapingSlashes es crítico aquí porque los uri llevan barras) — verificable extendiendo MCPEndToEndTests.
12. DADO el binario compilado y registrado con claude mcp add, CUANDO el usuario escribe / en Claude Code, ENTONCES los templates de la vault aparecen como comandos /pasture-mcp:… con sus argumentos — verificación MANUAL documentada en docs/mcp-server.md antes de anunciar la feature (riesgo declarado: soporte desigual de resources/prompts entre versiones de cliente; comprobar también @-mención de resources en Claude Desktop).

#### Criterios técnicos

- Nuevos archivos Sources/PastureKit/MCP/MCPResources.swift y MCPPrompts.swift siguiendo el patrón de MCPTools: enums públicos, Sendable, sin I/O de transporte propio; el executable main.swift NO cambia (ADR-MCP-004, el dispatcher sigue siendo la única frontera).
- MCPDispatcher.handle(line:) añade 4 cases al switch: resources/list, resources/read, prompts/list, prompts/get. Nunca lanza; todo fallo se traduce a línea de error o result con isError (SEC-M12).
- InitializeResult.Capabilities (MCPProtocol.swift) pasa de {tools} a {tools, resources, prompts}, los tres serializados siempre como objeto presente (test explícito de presencia de las 3 claves).
- Enumeración de resources y prompts EXCLUSIVAMENTE vía MCPTools.enumerateVaultFiles (FileLibrary: filtra ocultos y symlinks, raíz + 1 nivel). Cero código nuevo de enumeración de disco.
- resources/read: mapeo uri pasture:///<ruta-relativa> → MCPPathResolver.resolve (capa 1 PathValidator.isInside + capa 2 resolvingSymlinksInPath, SEC-M1/SEC-M2) ANTES de cualquier I/O; rechazo de esquemas distintos de pasture:// y de rutas absolutas.
- Caps heredados sin excepción: MCPLimits.maxInputLineBytes 10 MB en entrada (SEC-M3, ya lo aplica MCPLineReader sin cambios); MCPLimits.maxResponseBytes 25 MB en resources/read Y en el texto renderizado de prompts/get (SEC-M5, con pre-check fileSizeOnDisk en resources). Añadir MCPLimits.maxPromptArgumentLength (propuesta: 100.000 chars por valor de argumento) como constante pública testeada individualmente, coherente con el estilo de MCPLimits.
- prompts/get renderiza con TemplateEngine.render (single-pass; hereda maxNestingDepth=16 y maxIterations=1000 del motor) y pasa el resultado por SecretScanner.scan (cap 2 MB con corte en frontera UTF-8 válida, SecretScanner.cappedContent). Aclaración de invariantes: SEC-M6 ('el ensamblado de contexto va solo por ContextBuilder') sigue intacto — aplica a feed_context; prompts/get es render de template, canal distinto, y queda documentado así en CLAUDE.md.
- SEC-M11 preservado: las cuatro rutas nuevas son estrictamente read-only; el render ocurre en memoria, sin ninguna escritura a ~/.pasture/ ni a disco.
- SEC-M7 preservado: ningún print a stdout desde MCPResources/MCPPrompts; diagnóstico solo por stderr.
- Tests nuevos (Swift Testing, todos contra MCPDispatcher.handle(line:), sin spawn): suites MCPResourcesTests y MCPPromptsTests (~35-45 tests: catálogos, traversal, symlink, caps, defaults, required, inyección single-pass, secretos post-render con fixtures concatenados por Push Protection), ampliación de MCPProtocolTests (capabilities) y de MCPEndToEndTests (secuencia completa con las 3 primitivas). La suite total sube de 501; CI macos-15 sin cambios.
- Nombre de prompt = ruta relativa sin extensión .md con '/' sustituido (p. ej. proyecto__spec); el mapeo name→ruta se resuelve re-enumerando en prompts/get (sin estado entre requests, coherente con el loop secuencial ADR-MCP-005). Test de colisión: dos rutas que colisionen tras la sustitución → la segunda se descarta de prompts/list con log a stderr.
- Documentación: actualizar docs/mcp-server.md (tabla de primitivas, matriz de compatibilidad de clientes verificada manualmente), CLAUDE.md (secciones MCP layer y Security invariants), CHANGELOG.md (Keep a Changelog) y bump de MCPProtocol.serverVersion + VERSION en scripts/bundle.sh al liberar.

#### Dependencias

- Capa MCP v1.5.0 ya publicada: MCPDispatcher, MCPLineReader, MCPPathResolver, MCPLimits, mcpLine() (ADR-MCP-006) — base directa, sin cambios de transporte.
- TemplateEngine público en PastureKit: extractVariables(from:), render(_:with:), hasVariables/hasBlocks, VariableKind (.scalar/.list) — se consumen tal cual, sin tocar el motor.
- SecretScanner (v1.4.0) con scan(fileName:content:), summaryLines() y cappedContent — reutilizado para el guard post-sustitución (ADR-QW-002) y el warning enmascarado (SEC-M8/D4).
- MCPTools.enumerateVaultFiles + FileLibrary (filtrado de ocultos/symlinks) — única fuente de enumeración para resources y prompts.
- MCPServerConfig.fromEnvironment (ADR-MCP-007) — sin cambios; ninguna primitiva nueva lee UserDefaults.
- Verificación externa previa al anuncio: confirmar en qué versiones de Claude Code y Claude Desktop se consumen prompts (slash-commands) y resources (@-mención); si Claude Code no consume resources, la mitad prompts se anuncia sola (los jueces ya señalan que resources es la mitad marginal).
- Toolchain local: swift test requiere ~/Library/Developer/Toolchains/swift-latest.xctoolchain (el CLT no trae el módulo Testing).

#### Seguridad y privacidad

Superficie nueva acotada: las cuatro rutas son de solo lectura (SEC-M11 se conserva por diseño — sigue sin existir ningún write path a ~/.pasture/ en pasture-mcp) y cero dependencias externas (SEC-M10: nada nuevo en Package.swift). Los dos riesgos genuinamente nuevos son (1) valores de argumento controlados por el cliente entrando en el render — mitigado porque TemplateEngine es single-pass (un valor jamás se re-parsea como sintaxis, invariante ya testeado del motor) y porque el output renderizado pasa por SecretScanner post-sustitución (ADR-QW-002), con resumen siempre enmascarado (primeros 7 + últimos 4 chars, nunca el valor íntegro) entregado por el campo description de GetPromptResult como aviso informativo no bloqueante (SEC-M8/D4, misma política que read_file); y (2) el esquema de uri como nuevo vector de traversal — mitigado exigiendo esquema pasture:// exclusivo, rechazando rutas absolutas y pasando toda resolución por las dos capas de MCPPathResolver (SEC-M1 + SEC-M2, symlinks incluidos) antes de cualquier I/O. Decisión explícita a validar con el usuario: resources/read NO emite warning de secretos (el resultado estándar de resources carece de canal para ello); la cobertura la da el escaneo bajo demanda de Settings→MCP (SEC-M9, consent-first) — documentar esta asimetría en docs/mcp-server.md. Caps: entrada 10 MB/línea (SEC-M3), respuesta 25 MB (SEC-M5) también sobre el texto renderizado, argumentos de prompt con cap propio nuevo en MCPLimits, render limitado por maxNestingDepth=16 y maxIterations=1000. stdout sigue siendo sagrado (SEC-M7). Privacidad: nada sale de la máquina — el servidor solo responde por stdio al cliente local; ningún contenido se registra en logs (stderr solo lleva diagnóstico sin contenido de ficheros ni valores de argumentos).

### Context Compiler: packs de compilación del vault a CLAUDE.md y AGENTS.md con sync no destructivo

**Score del panel:** 7.8/10 · **Esfuerzo:** 3 semanas

**Qué es:** Un 'pack' de compilación = preset de selección + template renderizada (con variables por proyecto persistidas) + rutas destino en repos del usuario (CLAUDE.md, AGENTS.md, .cursor/rules/, .github/instructions/). Pasture compila y escribe los destinos con un click o al detectar cambios en los archivos fuente vía el DirectoryWatcher existente, con cabecera 'generated by Pasture' + hash del último sync para detectar ediciones manuales y nunca pisarlas sin confirmación. Un panel 'dónde está sincronizado esto' muestra el acoplamiento, y 'Sync all' desde la barra de menús propaga una corrección del vault a los N proyectos que la usan. Lanzamiento con 2 targets sólidos (CLAUDE.md, AGENTS.md) y emisores diseñados como adaptadores internos baratos de actualizar.

**Por qué es 10x:** La mejor relación esfuerzo/impacto del lote (5 de 5 perspectivas la propusieron): reposiciona Pasture de 'app que abres al pegar contexto' a infraestructura permanente — la fuente única de verdad de las instrucciones de TODOS los agentes (Claude Code, Cursor, Copilot, Codex) en todos los repos. Ataca la fricción nº1 del power user: la misma regla copiada en 15 CLAUDE.md que divergen en silencio. Es el nicho que Ruler/Packmind atacan a escala de organización y nadie cubre para el individuo en app nativa local. Es mayormente orquestación de piezas ya construidas (SelectionPreset, TemplateEngine, ContextBuilder, ExportDestination, DirectoryWatcher), y los archivos generados en repos compartidos son marketing pasivo ante colegas.

**Veredictos del panel (impacto / viabilidad / diferenciación):**

- Ataca un dolor real y recurrente del usuario objetivo: la divergencia silenciosa de CLAUDE.md/AGENTS.md entre repos es fricción que se paga cada semana, no una feature de un solo uso — cada corrección en el vault se propaga a N proyectos, lo que la convierte en multiplicador y reposiciona Pasture como infraestructura permanente en lugar de utilidad ocasional. El escepticismo aplica en dos frentes: el valor escala con el número de repos activos (un usuario con 2-3 proyectos apenas lo notaría, aunque el perfil power-user con 10+ repos sí lo notaría a diario), y las reglas compartidas no cambian tan a menudo como el pitch sugiere — el uso real será semanal/quincenal, no diario. Aun así, el panel de "dónde está sincronizado" y la garantía de no-divergencia aportan valor pasivo continuo incluso sin sincronizar. No es un 10 porque el coste de setup por pack (preset + variables + rutas destino) puede hacer que usuarios medios lo configuren una vez para 2 repos y ahí se quede.
- Es la candidata mejor alineada con la base real del repo: verificado en código que ExportDestination/exportToFile ya escriben fuera del vault como patrón documentado, DirectoryWatcher expone un onChange debounced reutilizable, y TemplateEngine/SelectionPreset/ContextBuilder son piezas puras de PastureKit listas para orquestar. No exige dependencias externas (hash vía CryptoKit), no hay backend, y no toca el MCP read-only (la escritura vive en la GUI). Las 3 semanas son plausibles aunque justas: el modelo de no-destrucción (marcador+hash+confirmación) es lógica pura testeable con la suite existente, pero la UX de conflictos y el decode retrocompatible de presets pueden comerse margen. Descuento medio punto por la deuda perpetua de adaptadores de formatos ajenos (.mdc de Cursor) y porque el auto-resync escribiendo en árboles git del usuario es el primer write-path automático de la app — bien mitigado, pero es donde un bug destruye la confianza.
- Los emisores en sí (escribir CLAUDE.md/AGENTS.md desde una fuente) los copia cualquiera en un fin de semana, y de hecho ya existen gratis en CLI (Ruler, rulesync y afines) para el mismo usuario objetivo que vive en la terminal — eso limita el techo. Pero la propuesta no compite en el emisor sino en la composición, que sí explota la posición única de Pasture: el vault ya es la fuente de verdad viva del usuario, y presets + TemplateEngine con variables por proyecto + DirectoryWatcher para auto-resync + panel de acoplamiento + menú bar es una experiencia integrada local-first que un CLI no replica y que un competidor solo copia si ya tiene el gestor de vault entero. Además refuerza el moat existente (cuanto más contexto vive en Pasture, más caro es salir) y los archivos generados son marketing pasivo. El riesgo diferencial real es el mantenimiento perpetuo de formatos ajenos (.mdc, convenciones AGENTS.md), que erosiona la ventaja si se descuida.

#### User stories

- Como power user con muchos repos, quiero definir un 'pack' (preset de selección + template con variables por proyecto + rutas destino en mis repos) para que mis reglas comunes vivan una sola vez en ~/.pasture/ y se compilen a cada CLAUDE.md/AGENTS.md sin copiar-pegar.
- Como usuario que edita a mano algún CLAUDE.md generado, quiero que Pasture detecte mi edición manual (vía hash del último sync) y me pida confirmación explícita antes de sobrescribir, para no perder nunca trabajo hecho fuera de Pasture.
- Como usuario que corrige una regla en el vault, quiero un botón 'Sync all' (en ventana principal y barra de menús) que propague el cambio a los N destinos de todos los packs, para arreglar en un sitio y desplegar en todos.
- Como usuario, quiero un panel 'dónde está sincronizado esto' por archivo del vault y por pack (destino, fecha del último sync, estado: al día / desfasado / conflicto / destino ausente), para ver el acoplamiento antes de tocar nada.
- Como usuario prudente, quiero que cada escritura conserve un backup del contenido anterior del destino y que el escaneo de secretos corra sobre el contenido compilado antes de escribir, para que un sync jamás destruya ni filtre nada.
- Como usuario avanzado, quiero activar auto-resync opt-in por pack (aprovechando el watcher de ~/.pasture/) para que un cambio en los archivos fuente recompile los destinos sin abrir la app — pero sin sobrescribir jamás un destino en conflicto.
- Como usuario de varias herramientas (Claude Code, Codex), quiero que un mismo pack emita a targets distintos (CLAUDE.md, AGENTS.md) mediante adaptadores internos, para añadir formatos futuros (.cursor/rules, .github/instructions/) sin rediseñar el modelo.

#### Criterios de aceptación

1. Given un pack con preset de 2 archivos del vault, template con variable {{PROJECT}} persistida con valor 'foo' y target CLAUDE.md en un directorio temporal, When se ejecuta 'Compile & write', Then el archivo destino existe, empieza por la cabecera de marcador de Pasture (nombre de pack + hash SHA-256 del cuerpo + aviso 'generated by Pasture — do not edit'), y su cuerpo es exactamente la salida de TemplateEngine.render + ContextBuilder.build para esos archivos con PROJECT=foo (verificable con test unitario de PackCompiler comparando strings).
2. Given un destino previamente escrito por Pasture cuyo cuerpo fue editado a mano después, When se pide sync de ese pack, Then el hash del cuerpo actual no coincide con el registrado, el sync de ese destino se detiene con estado 'conflicto', se muestra diálogo con opción por defecto Cancel, y el archivo NO se modifica salvo confirmación explícita 'Overwrite' (test: alterar el cuerpo tras escribir, reintentar sync, verificar contenido intacto sin confirmación).
3. Given un destino en conflicto y confirmación 'Overwrite' del usuario, When se sobrescribe, Then existe un backup del contenido anterior en ~/Library/Application Support/Pasture/backups/ con timestamp, nunca dentro del repo del usuario ni de ~/.pasture/, y se conservan como máximo los 10 últimos backups por destino (test: comprobar creación del backup y la poda al undécimo sync).
4. #### Seguridad y privacidad

Este feature introduce el primer write-path de Pasture hacia árboles git del usuario, así que el modelo de amenaza cambia: el riesgo principal ya no es exfiltración sino destrucción de trabajo ajeno a Pasture. Mitigaciones de diseño: sync estrictamente unidireccional (vault → destinos; el conflicto bidireccional se detecta por hash y se rechaza, nunca se mergea), marcador + SHA-256 del cuerpo para distinguir 'lo escribió Pasture' de 'lo tocó el humano', default Cancel en todo diálogo de sobrescritura, backups fuera del repo del usuario (en Application Support, para no ensuciar su git ni su vault), escritura atómica, y un destino sin marcador de Pasture se trata SIEMPRE como conflicto (protege el caso de apuntar por error a un CLAUDE.md preexistente escrito a mano). Privacidad: los packs persisten solo referencias (UUID de preset, rutas, variables) — jamás contenido de archivos ni credenciales; las variables por proyecto se guardan en UserDefaults en claro, así que la UI debe advertir que no se pongan secretos en variables, y el SecretScanner post-render actúa como red de seguridad antes de cada escritura (los archivos generados acaban commiteados en repos potencialmente públicos — este es el vector de fuga más plausible). El servidor MCP permanece read-only (SEC-M11 intacto): ningún tool nuevo, la compilación vive solo en la GUI. Si la app se sandboxea en el futuro, los targets necesitarán security-scoped bookmarks (mismo caveat ya documentado para ExportDestination). CryptoKit es framework de Apple: se mantiene la regla de cero dependencias externas.

## Fase 2 — v1.7 «Memoria viva»

**Por qué esta fase:** El vault deja de ser una biblioteca estática: los agentes proponen conocimiento (nunca escriben directo) y el contexto declara su propia frescura. Bloque fundacional compartido: FrontmatterParser.

### Contexto vivo: caducidad declarada + fuentes locales re-importables (Pasture v1.6)

**Score del panel:** 7/10 · **Esfuerzo:** 3.5 semanas

**Qué es:** Frontmatter con dos mecanismos: caducidad (review_after / ttl) con badges de frescura en sidebar, cola de revisión que agrupa lo vencido para repasarlo en un minuto, y anotación de staleness en las respuestas MCP ('este archivo lleva 120 días sin revisión') para que el agente consumidor sepa cuánto fiarse; y fuentes (source: carpeta local cuyo contenido se re-importa al abrir la app), con la nota marcada como generada/solo-lectura. Versión recortada deliberadamente: sin URLs ni ejecución de comandos en v1 — solo carpetas locales, que reduce la superficie de seguridad a casi cero.

**Por qué es 10x:** Ataca el problema silencioso nº1 de todo vault de contexto: la memoria podrida. Un contexto en el que se puede confiar es la condición para delegar más en agentes; sin frescura verificable el vault degrada solo y el usuario deja de usarlo (el ciclo de muerte de todo PKM). La señal de staleness expuesta por MCP es única en el mercado: los agentes pueden ponderar la fiabilidad de lo que leen. Sinergia directa con Context Health (comparten la noción de 'rancio': allí inferida por uso, aquí declarada por el autor).

**Veredictos del panel (impacto / viabilidad / diferenciación):**

- La caducidad ataca un dolor real y recurrente (la podredumbre silenciosa del vault) y la anotación de staleness vía MCP es lo único con potencial de multiplicador: convierte el vault en contexto con fiabilidad medible para agentes, que es exactamente el caso de uso diario de Pasture. Pero el valor está condicionado a la disciplina del usuario — quien no mantiene su vault tampoco declarará review_after en frontmatter, así que la feature es inerte justo para quien más la necesita, y los badges degradan a ruido visual si la cola se ignora (riesgo que la propia candidata admite). La mitad de fuentes re-importables es un nice-to-have de frecuencia media que Pasture ya cubre parcialmente con el DirectoryWatcher y la edición externa; empaquetarla aquí diluye el foco de las 2,5 semanas.
- Encaja muy bien con la base verificada: ToolCallResult ya tiene el canal `warning` no bloqueante (SEC-M8) donde la anotación de staleness entra sin romper el modelo read-only del MCP; la re-importación de carpetas locales vive en la GUI, que ya escribe en el vault (scanFolder/DocumentImporter existen), así que ningún invariante de seguridad se toca — solo hay que validar la ruta de la fuente y respetar isInsidePasture en destino. El coste real oculto es que hoy NO existe parser de frontmatter (MDFile es un struct plano) y YAML completo exigiría dependencia: hay que hand-rollear un subconjunto key:value al estilo TemplateEngine, viable pero es trabajo nuevo con suite de tests exigente (491 tests) más la decisión de si el frontmatter se stripea en ContextBuilder. Por eso 2,5 semanas es optimista para las dos mitades juntas (frescura + fuentes con marca de solo-lectura e interacción con DirectoryWatcher); 3,5-4 sería más honesto, o recortar a solo la mitad de caducidad+MCP, que sí cabe en ~1,5.
- La mitad "caducidad" es commodity: badges de frescura, review_after en frontmatter y cola de revisión son replicables en un fin de semana y ya existen aproximaciones gratis en Obsidian (Dataview + plugins de review) que el usuario objetivo probablemente usa. Lo que sí explota la posición única de Pasture es la anotación de staleness en las respuestas MCP: exige poseer a la vez el vault, la UI donde el autor declara la caducidad y el servidor MCP que la sirve al agente — esa cadena completa (nativo, local-first, MCP integrado) no la tiene ningún competidor hoy, y es una señal que ningún gestor de contexto expone. La parte de fuentes locales re-importables es genérica (symlinks, importers, sync) y aporta poco a la diferenciación; diluye más que suma bajo esta lente.

#### User stories

- Como autor del vault, quiero declarar en frontmatter la caducidad de una nota (`review_after: 2026-09-01` o `ttl: 90d`) para que Pasture sepa cuándo deja de ser fiable sin que yo tenga que recordarlo.
- Como usuario de la app, quiero ver badges de frescura en el sidebar (fresca / vencida) para detectar de un vistazo el contexto podrido antes de alimentárselo a un agente.
- Como usuario, quiero una cola de revisión que agrupe todas las notas vencidas y me permita marcar cada una como revisada en un clic (escribiendo `last_reviewed` en su frontmatter) para sanear el vault en un minuto.
- Como agente que consume el vault vía pasture-mcp, quiero que `read_file` y `feed_context` anoten la antigüedad de revisión de cada fichero vencido ('stale: 120 days since last review') para ponderar cuánto fiarme de ese contexto.
- Como usuario con documentación viva en otra carpeta local (p. ej. docs de un repo), quiero declarar `source: <carpeta>` en una nota para que Pasture re-importe su contenido .md al abrir la app o al pulsar Refrescar, sin copiar a mano.
- Como usuario, quiero que las notas producidas por una fuente lleven la marca `generated: true` y se muestren como solo-lectura en la UI, para saber que una re-importación las sobrescribirá y no perder ediciones manuales por sorpresa.
- Como usuario que no usa caducidad ni fuentes, quiero que la feature sea totalmente inerte (cero badges, cero cola, cero escaneos extra) para que mi flujo v1.5 no cambie en nada.

- Fase B (fuentes re-importables) depende de la Fase A (parser + marca `generated` en frontmatter). Recomendación de secuenciación: Fase A = caducidad + badges + cola + MCP staleness (~1,5 sem, entregable por sí sola); Fase B = fuentes locales (~2 sem).

- Toolchain local: `swift test` requiere el toolchain de `~/Library/Developer/Toolchains/swift-latest.xctoolchain` (el CLT no trae el módulo Testing) — restricción de entorno conocida, no de producto.

#### Seguridad y privacidad

Superficie de ataque deliberadamente mínima: v1 solo admite carpetas LOCALES como fuente — sin URLs, sin ejecución de comandos, sin red (si algún día entran, será con allowlist y consentimiento por fuente; hoy quedan explícitamente fuera de alcance). El servidor MCP conserva su garantía central: sigue sin ningún write path a ~/.pasture/ (SEC-M11) — tanto la escritura de last_reviewed como la re-importación viven solo en la GUI; la señal de staleness viaja por el canal warning informativo (SEC-M8) y jamás altera contenido ni filtra valores (compatible con el enmascaramiento de SecretScanner). Validación de fuentes: ruta resuelta con resolvingSymlinksInPath(), rechazo de fuentes dentro del vault (anti-ciclo de re-importación) y destino siempre validado con isInsidePasture() antes de escribir. Anti-DoS del parser: frontmatter capado a 8 KB/64 líneas, sin regex vulnerables a backtracking, claves desconocidas ignoradas en silencio — un frontmatter hostil nunca tumba la app ni el servidor MCP (un fallo de parseo degrada a 'sin caducidad', nunca a error de protocolo, SEC-M12). Privacidad: los metadatos de frescura (fechas) no son sensibles y no salen de la máquina salvo dentro del propio warning MCP local por stdio; la re-importación no realiza ninguna llamada de red y el contenido importado pasa por el mismo escaneo de secretos pre-feed que el resto del vault. Riesgo residual asumido y documentado: una re-importación sobrescribe ediciones manuales de notas generated: true — mitigado porque la marca es visible en la UI, retirarla del frontmatter desvincula la nota de su fuente, y la decisión sobrescribir/preservar está cubierta por test.

#### Seguridad y privacidad

El SecretScanner pre-feed no cambia: sigue escaneando el contexto que se envía; extenderlo al historial multi-turno queda explícitamente fuera de alcance de esta iteración (el historial es output del modelo + preguntas del usuario, no contenido de archivos), y se documenta como límite conocido.

## Fase 4 — v1.9 «Observabilidad»

**Por qué esta fase:** Auditabilidad y reproducibilidad: receipts de cada entrega, snapshots con historial/diff/restore y panel de salud. Es lo que evita que las fases anteriores colapsen el vault en un desván.

### Context Health + procedencia: feed receipts append-only, snapshots content-addressed con historial/diff/restore, panel de salud por archivo y tool MCP get_feed_history

**Score del panel:** 6.8/10 · **Esfuerzo:** 6 semanas

**Qué es:** Cada entrega de contexto (clipboard, export, Ask, MCP) registra un receipt local: archivos con hash de contenido, canal, formato, tokens, timestamp — en formato append-only seguro ante escrituras concurrentes de GUI y proceso MCP (ficheros por-proceso + merge en lectura). Los contenidos se snapshotean por hash en ~/.pasture/.history/ (dedupe automático) con GC/retención configurable. Dos superficies en la GUI: panel de salud por archivo (frecuencia de uso, último feed, alertas de 'contexto muerto' — 0 usos en 60 días — y 'contexto rancio' — muy usado sin editar en meses) y vista Historial (línea temporal, diff entre versiones, restaurar). Tool MCP get_feed_history para que un agente consulte qué contexto recibió en sesiones anteriores.

**Por qué es 10x:** Auditabilidad y reproducibilidad son la línea que separa una carpeta de .md de infraestructura: poder responder '¿qué vio exactamente el agente ayer?' convierte a Pasture en la herramienta de depuración de comportamiento de agentes que no existe en local — lo que PromptLayer/Langfuse venden como versionado, sin cloud, sin cuenta, sin telemetría. Y es lo que evita que las demás features colapsen el vault: sin señal de uso, 100+ archivos son un desván donde nadie sabe qué borrar; el rollback por archivo elimina el miedo a editar contexto. Los hashes son la base sobre la que se apoyan el Memory Inbox y la caducidad. Diferenciación pura de identidad: 100% local y borrable.

**Veredictos del panel (impacto / viabilidad / diferenciación):**

- La mitad "salud" (frecuencia de uso, último feed, alertas de contexto muerto/rancio) sí tocaría al usuario diario de forma pasiva y resuelve un dolor real que crece con el vault — es la única señal que dice qué borrar. Pero la mitad "procedencia" (snapshots, diff, restaurar, get_feed_history) es valor de consulta ocasional, no diaria: "¿qué vio el agente ayer?" se pregunta unas pocas veces al mes, y el versionado/rollback lo cubre hoy un `git init` en ~/.pasture o Time Machine, con lo que 4,5 semanas compran mucho andamiaje para poco uso activo. Además hay cold-start: las alertas de 60 días no muestran nada útil hasta pasados meses de receipts acumulados. Es infraestructura habilitadora más que multiplicador directo — valdría más una versión recortada (solo receipts ligeros + panel de salud, sin snapshots ni diff) a ~1,5 semanas.
- Construible al 100% con la base actual y sin romper la regla de cero dependencias: hashing con CryptoKit (framework de Apple), receipts/snapshots con Foundation, y verifiqué que FileLibrary usa .skipsHiddenFiles en sus tres enumeraciones, así que ~/.pasture/.history/ no contaminaría la biblioteca ni las tools MCP. La tensión con SEC-M11 (pasture-mcp sin write path al vault) está identificada por la propia candidata y tiene mitigación limpia — receipts solo del lado GUI en fase 1, y get_feed_history es read-only puro, encajando en el patrón MCPTools existente. El punto débil es el esfuerzo: 4,5 semanas es optimista para receipts append-only con merge multi-proceso + snapshots con GC + dos superficies GUI (panel de salud y timeline con diff/restore, incluyendo un diff Myers artesanal al no poder usar librerías) + tool MCP + la cultura de tests del proyecto (~500 tests); 6-7 semanas es más realista, o bien recortar el diff visual de la fase 1.
- Alta diferenciación estructural: los feed receipts nacen de datos que solo Pasture posee (los eventos de entrega por clipboard/export/Ask/MCP), así que un competidor no lo copia en un fin de semana sin tener antes esa superficie de entrega instrumentada — y explota directamente los cuatro pilares de identidad (local-first, sin cuenta, MCP integrado con get_feed_history, privacidad borrable), atacando lo que Langfuse/PromptLayer solo ofrecen con cloud y cuenta. La señal de 'contexto muerto/rancio' tampoco existe gratis en las herramientas del usuario objetivo (git versiona contenido, no uso ni entregas). El punto débil es la mitad de snapshots/diff/restore: eso sí lo cubren parcialmente git, Time Machine o el File Recovery de Obsidian, por lo que la diferenciación real vive en receipts + salud de uso, no en el versionado.

#### Dependencias

- FeedService como ruta única de entrega (ya existente desde la auditoría 2.1: DeliveryTarget unifica clipboard/destination/fileURL) — el receipt se cuelga de deliverFeed; si apareciera una nueva vía de entrega que no pase por ahí, rompe la completitud de los receipts.

#### Seguridad y privacidad

Todo es 100% local y borrable: ningún dato sale de la máquina, no hay telemetría ni cuenta, y eliminar ~/.pasture/.history/ devuelve la app al estado pre-feature sin residuos (criterio de aceptación explícito). Los receipts contienen solo metadatos (rutas relativas, hashes SHA-256, canal, formato, tokens, timestamps) — nunca contenido, secretos, maskedSnippets ni API keys. Los snapshots SÍ contienen el contenido entregado (esa es su función): heredan la superficie de riesgo del vault, viven bajo el mismo directorio protegido por PathValidator, quedan fuera del alcance de las tools MCP de lectura de vault (ocultos) y su exposición vía get_feed_history está prohibida (la tool devuelve metadatos, jamás contenido de snapshot). Tensión principal: SEC-M11 (pasture-mcp sin write path) — se resuelve por fases: Fases 1-2 el proceso MCP no escribe nada (receipts solo del lado GUI); Fase 3, si el usuario la aprueba, la escritura se acota a .history/receipts/ con validación de path y se documenta en CLAUDE.md como excepción explícita al modelo read-only. El restore es la única operación de escritura nueva sobre el vault y pasa por las dos capas de validación (PathValidator.isInside + resolvingSymlinksInPath, patrón MCPPathResolver) para que un receipt manipulado a mano nunca escriba fuera de ~/.pasture/. Riesgo de percepción: la métrica se comunica como 'uso' (no 'calidad' ni 'tracking'), con copy en la UI que subraya local + borrable. Riesgo de disco: GC con defaults conservadores (90 días / 500 MB) y caps de snapshot (5 MB/archivo) activos desde el primer commit, no como mejora posterior.

## Fase 5 — v2.0 «Inteligencia on-device»

**Por qué esta fase:** Selección de contexto por relevancia con presupuesto de tokens, 100% on-device (NaturalLanguage + BM25). Condicionada a un spike go/no-go de 2-3 días que puede ejecutarse en paralelo durante la Fase 2.

### Smart Select: motor de relevancia on-device con presupuesto de tokens (GUI + MCP)

**Score del panel:** 7.2/10 · **Esfuerzo:** 7 semanas

**Qué es:** Índice semántico local con NLEmbedding/NLContextualEmbedding del framework NaturalLanguage de Apple (cero dependencias) combinado con ranking léxico BM25 en Swift puro; troceado + agregación por documento, caché en ~/.pasture/.index/ invalidada por el DirectoryWatcher. Sobre él, tres superficies: (1) 'Select for task' — describes la tarea y un presupuesto de tokens y Pasture pre-marca la selección óptima con explicación por archivo, editable antes del Feed y guardable como 'smart preset' (guardas la intención, no la lista fija); (2) búsqueda semántica en sidebar y en la tool MCP search con fallback literal; (3) tool MCP select_context(task, budget) para que Claude Code componga su propio contexto documental. Motor heurístico instantáneo por defecto; refinado opcional vía el Ask existente (solo títulos+extractos, opt-in explícito).

**Por qué es 10x:** Ataca la presión competitiva nº1 (Context Builder de Repo Prompt, la tendencia entera de curación agéntica): el modelo manual de checkboxes tiene deseconomía de escala — cuanto más crece el vault, peor la UX. Esto invierte la curva: vault más grande = mejores sugerencias, la definición de moat personal. Y lo hace con el ángulo que nadie puede copiar sin renunciar a su arquitectura: 100% on-device, sin API key, sin red — solo una app nativa Swift accede a estos frameworks. Vía MCP, el vault pasa de listado paseable a memoria recuperable por relevancia: Claude Code pide 'el mejor contexto sobre X en 20K tokens' y lo recibe.

**Veredictos del panel (impacto / viabilidad / diferenciación):**

- Toca la acción central y más repetida del producto — decidir qué archivos entran al feed — así que si funciona es un multiplicador de uso diario, no un adorno: cada sesión de trabajo pasa por ahí, y la tool MCP select_context convierte el vault en memoria consultable para Claude Code, un caso de uso genuinamente recurrente. Pero el valor es condicional dos veces: solo aparece cuando el vault crece lo bastante para que los checkboxes duelan (en vaults de 20-40 archivos los presets existentes ya resuelven el 80% del problema), y depende críticamente de la calidad de NLEmbedding — dos sugerencias malas y el usuario vuelve al modo manual, dejando 6 semanas de trabajo como un botón ignorado. El diseño mitiga bien (sugerencia revisable, fallback BM25, spike go/no-go), pero la frecuencia real de uso hoy es menor de lo que sugiere el pitch competitivo.
- Construible sobre la base actual: NaturalLanguage es framework de Apple (macOS 14+, cero deps preservado), BM25 en Swift puro es trivial, y la capa MCP existente admite un tool nuevo siguiendo el patrón de MCPTools; el propio pitch nombra las mitigaciones correctas (spike go/no-go, fallback léxico, sugerencia revisable, refinado opt-in vía Ask). Pierde puntos por dos flancos: (1) la caché en ~/.pasture/.index/ crea tensión con SEC-M11 — si el proceso pasture-mcp escribe o refresca el índice rompe el invariante read-only; es mitigable (GUI indexa, MCP lee o computa efímero) pero la candidata no lo resuelve explícitamente, y de paso read_file podría exponer los blobs del índice; (2) 6 semanas es el esfuerzo mayor del lote y está en el borde optimista para tres superficies + indexado en background bajo Swift 6 strict concurrency + eval set, con la mayor incertidumbre técnica (calidad de NLEmbedding en español/contenido técnico) aún sin validar.
- Explota de verdad la posición única: solo una app nativa Swift accede a NLEmbedding/NLContextualEmbedding sin dependencias ni red, y "relevancia semántica 100% on-device, sin API key" es un titular que Repo Prompt (Electron-adjacent) o cualquier wrapper web no puede replicar sin rehacer su arquitectura — y con caché, invalidación por watcher, BM25 híbrido y tres superficies (GUI + sidebar + MCP), no es copiable en un fin de semana. Además convierte la desventaja de escala del modelo checkbox en moat (vault más grande = mejores sugerencias) y compone con el activo MCP existente. Descuento por dos motivos: el ingrediente diferencial (embeddings de Apple) es a la vez el más débil técnicamente — si el spike falla, queda BM25, que sí es commodity copiable —, y la parte MCP compite con algo que el usuario objetivo ya tiene gratis: la búsqueda agéntica de Claude Code sobre el propio vault (grep + read via las tools actuales) ya resuelve una fracción del caso "componme el contexto sobre X".

#### Seguridad y privacidad

Privacidad: todo el pipeline de relevancia (troceado, embeddings NL, BM25, empaquetado) corre on-device; la app no abre ninguna conexión de red para Smart Select — la única red posible es la descarga de assets de NLContextualEmbedding, gestionada por el sistema operativo, y debe señalizarse en la UI de primer indexado. El refinado vía Ask es opt-in explícito con diálogo de consentimiento (patrón askPrivacyNoticeAccepted) y envía únicamente títulos + extractos ≤200 caracteres, nunca archivos completos; con el opt-in apagado, tests verifican cero invocaciones de AIClient desde Smart Select. El punto más delicado es la caché ~/.pasture/.index/: (1) resuelve la tensión con SEC-M11 con una regla de escritura asimétrica — solo la GUI escribe el índice; el proceso pasture-mcp lo lee o computa BM25 efímero en memoria, y un test E2E verifica que el árbol de ~/.pasture/ es idéntico antes y después de servir select_context; (2) los blobs del índice contienen derivados del contenido (términos, frecuencias, vectores) que pueden filtrar información, así que .index/ queda excluido de list_files/read_file/search/feed_context (ya invisible por el filtrado de ocultos de FileLibrary.visibleContents, más rechazo explícito en MCPPathResolver como defensa en profundidad) y NUNCA almacena valores de secretos — se indexa el contenido ya pasado por el cap de 2 MB con corte UTF-8 seguro. SmartPreset persiste solo {nombre saneado, texto de tarea, presupuesto}: ni contenido, ni URLs absolutas, ni API keys (herencia de ADR-QW-003/SEC-7). Los warnings de secretos en select_context siguen SEC-M8 (familia + archivo + maskedSnippet, informativo, no gate). Sin dependencias nuevas en Package.swift (SEC-M10 intacto): NaturalLanguage es framework de Apple. Superficie DoS acotada por los caps nuevos (task 1.000 chars, budget ≤ 2M, 100 archivos, índice ≤ 200 MB con purga) y los existentes de MCPLimits.
