# PRD: v1.4 -- Quick wins

**Producto:** Pasture (macOS)
**Version objetivo:** 1.4.0
**Fecha:** 2026-06-12
**Autor:** El Buscador de Problemas (PO)
**Estado:** BORRADOR -- pendiente de aprobacion

---

## 1. Problema

Pasture v1.3.0 ya hace bien lo nuclear: curar Markdown de contexto y alimentarlo a un AI (clipboard, export, Ask). Pero el flujo diario del usuario tipo -- un desarrollador que usa Claude Code/Desktop y quiere alimentar contexto curado rapido -- choca con cuatro fricciones concretas, ninguna de ellas resuelta hoy:

### 1A. No hay red de seguridad ante fugas de secretos

Hoy, cuando el usuario selecciona ficheros y hace Feed, Pasture copia/exporta/envia el contenido **tal cual**, sin inspeccionarlo. Si uno de esos ficheros contiene una clave de API (`sk-ant-...`, `ghp_...`), una credencial AWS o una clave privada PEM, esa credencial acaba en el portapapeles, en un fichero exportado o, peor, enviada a la API de un tercero (Anthropic/OpenRouter) sin que el usuario se de cuenta.

**Dolor concreto:** El usuario que pega un `.env` o un log a la carpeta de contexto y lo alimenta sin revisar filtra un secreto sin saberlo. No hay aviso, no hay friccion, no hay vuelta atras una vez enviado a la API.

### 1B. Recomponer la misma seleccion de ficheros, una y otra vez

El usuario trabaja en "proyecto X" y para ese proyecto siempre alimenta los mismos 5 ficheros de contexto. Cada vez que vuelve, tiene que volver a marcarlos uno a uno -- en la ventana principal y, por separado, en el menu bar (las selecciones son independientes). Es la friccion mas repetida del usuario tipo: trabajo manual que se repite identico cada sesion.

**Dolor concreto:** No hay forma de guardar una combinacion de ficheros con nombre y alimentarla de un clic. La seleccion es efimera y hay que reconstruirla cada vez.

### 1C. El feed puede exceder el contexto del modelo sin avisar

En el panel Ask existe el indicador `~Xk / Yk tokens` que compara la seleccion con el contexto del modelo configurado. Pero al hacer **Feed** (clipboard/export), el sidebar solo muestra `~Xk tokens` -- sin denominador, sin aviso. El usuario puede copiar una seleccion de 300k tokens para un modelo de 200k y solo descubrir el problema cuando el AI rechaza o trunca el contexto.

**Dolor concreto:** El usuario no sabe, antes de alimentar, si su seleccion cabe en el modelo que va a usar. El dato existe en Ask pero no en el flujo de Feed.

### 1D. Un unico formato de salida (XML+CDATA) no encaja con todos los destinos

El feed siempre sale como `<context name="..."><![CDATA[...]]></context>`. Es robusto para parsing por modelos, pero no es lo que algunas herramientas o flujos esperan. Un usuario que pega en un chat web, en un README o en un sistema que prefiere Markdown plano quiere `## filename` + bloques de codigo con fences. El formato XML es ruido para ese caso.

**Dolor concreto:** El usuario no puede elegir como sale su contexto. El XML+CDATA es la unica opcion, y para muchos destinos es el formato equivocado.

## 2. Contexto

- Pasture v1.3.0 esta publicada. Arquitectura de dos targets: PastureKit (logica testeable, 327 tests con Swift Testing) y Pasture (UI).
- **Cero dependencias externas.** Solo frameworks de Apple. Esta restriccion se mantiene y condiciona el alcance: nada de tiktoken, Secretlint, repomix ni librerias de terceros. Los patrones de deteccion de secretos son **propios** (regex/literales en Swift puro).
- Referencias de mercado (informe del Detective): repomix y Secretlint (deteccion de secretos), PasteMax (presets/workspaces + aviso de limite de contexto), files-to-prompt (formatos de salida). Son referencias de *que* problema resuelven, no de *como* lo implementan.
- Aviso del Detective: el competidor `prompt_builder` murio de "todo en uno". **El alcance de v1.4 esta acotado a estos 4 puntos.** Cualquier cosa fuera de ellos es feature creep y se rechaza.
- Puntos de anclaje en el codigo actual (verificados):
  - `ContextBuilder.build(files:)` en PastureKit es el **unico** punto de generacion del feed. Las features 1 (escaneo) y 4 (formatos) se anclan aqui.
  - `ExportFileFormat` (enum `.markdown`/`.plainText` en `ExportSettings.swift`) es el patron ya existente para un picker de formato en Settings. La feature 4 lo reutiliza/extiende.
  - El indicador `~Xk / Yk tokens` de Ask vive en `AskView.swift` (linea ~69) usando `TokenEstimator.formatted` + `model.contextWindow`. El `selectionSummary` del sidebar (`SidebarView.swift` ~241-263) ya calcula `totalTokens` con `TokenEstimator.formatted`. La feature 3 es, casi entera, anadir el denominador y un aviso.
  - `FeedService.deliverFeed(...)` es el punto compartido entre ventana principal y menu bar donde se entrega el feed. Las features 1 y 2 enganchan aqui.

## 3. Solucion propuesta (alto nivel)

> El detalle de implementacion es responsabilidad del architect y del senior-dev. Aqui solo se fija el QUE y las decisiones de producto.

- **F1 -- SecretScanner:** un tipo nuevo en PastureKit (`SecretScanner`) con un catalogo de patrones propios (literales + regex sin dependencias). Se invoca en `FeedService` antes de entregar el feed (clipboard, export y Ask). Si detecta algo, **avisa** -- no bloquea (ver decision en casos borde).
- **F2 -- Presets de seleccion:** persistir combinaciones de ficheros con nombre. Guardar la seleccion actual como preset; aplicar un preset con un clic desde la ventana principal y desde el menu bar. Persistencia en UserDefaults (mismo patron que ExportSettings/AISettings), referenciando ficheros por path relativo a `~/.pasture/`.
- **F3 -- Aviso de limite de contexto en el feed:** extender el `selectionSummary` del sidebar para mostrar `~Xk / Yk tokens` (denominador = `model.contextWindow` del modelo configurado en AISettings) y un aviso visual cuando la seleccion excede el contexto.
- **F4 -- Formatos de salida adicionales:** ademas del XML+CDATA actual, anadir formato **Markdown** (`## filename` + bloques con fences) y **texto plano** (separadores simples). Picker en Settings (mismo patron que ExportFileFormat). El formato aplica al feed (clipboard + export + contexto de Ask).

## 4. Historias de usuario priorizadas

Prioridad por relacion valor/riesgo y por dependencia con las security invariants del proyecto.

| # | Feature | Prioridad | Justificacion |
|---|---------|-----------|---------------|
| P0 | F1 -- SecretScanner | **Critica** | Previene fuga de credenciales. Encaja con las security invariants. 100% testeable en PastureKit. Es la unica con impacto de seguridad real. |
| P1 | F2 -- Presets de seleccion | **Alta** | Resuelve la friccion mas repetida del usuario tipo. Mayor impacto en uso diario. |
| P2 | F3 -- Aviso de limite de contexto | **Media** | Casi solo UI, bajo riesgo, valor claro. El dato ya existe; falta exponerlo en el flujo de Feed. |
| P3 | F4 -- Formatos de salida | **Media** | Aditivo y acotado. Reutiliza un patron existente. Menor urgencia que las anteriores. |

### F1 -- Deteccion de secretos antes del feed (P0)

**HU-1: Avisar de secretos antes de copiar/exportar**
Como desarrollador que alimenta contexto a un AI, quiero que Pasture inspeccione los ficheros seleccionados en busca de credenciales antes de copiarlos al portapapeles o exportarlos, para no filtrar un secreto sin darme cuenta.

**HU-2: Avisar de secretos antes de enviar a la API (Ask)**
Como desarrollador que usa el panel Ask, quiero que Pasture me avise si el contexto que voy a enviar a la API de Anthropic/OpenRouter contiene credenciales, para evitar enviar un secreto a un tercero de forma irreversible.

**HU-3: Decidir si continuar pese al aviso**
Como desarrollador que sabe que el "secreto" detectado es un ejemplo de documentacion (falso positivo), quiero poder continuar con el feed pese al aviso, para no quedar bloqueado por una deteccion que no aplica a mi caso.

### F2 -- Presets de seleccion (P1)

**HU-4: Guardar la seleccion actual como preset**
Como desarrollador que trabaja en varios proyectos, quiero guardar mi seleccion actual de ficheros con un nombre ("proyecto X"), para no tener que volver a marcarlos uno a uno en cada sesion.

**HU-5: Aplicar un preset desde la ventana principal**
Como desarrollador que retoma un proyecto, quiero aplicar un preset guardado de un clic desde la ventana principal, para reconstruir mi seleccion de contexto al instante.

**HU-6: Aplicar un preset desde el menu bar**
Como desarrollador que quiere un feed rapido sin abrir la ventana principal, quiero aplicar un preset desde el menu bar, para alimentar mi contexto curado sin salir de la barra de menus.

**HU-7: Gestionar presets (renombrar / eliminar)**
Como desarrollador cuyos proyectos cambian, quiero renombrar y eliminar presets, para mantener mi lista de presets util y sin ruido.

### F3 -- Aviso de limite de contexto en el feed (P2)

**HU-8: Ver cuanto contexto cabe en el modelo al seleccionar**
Como desarrollador que va a hacer Feed, quiero ver `~Xk / Yk tokens` en el resumen de seleccion del sidebar (igual que ya veo en Ask), para saber cuanto margen me queda respecto al modelo configurado.

**HU-9: Recibir un aviso cuando la seleccion excede el contexto**
Como desarrollador que ha seleccionado demasiados ficheros, quiero un aviso visual claro cuando mi seleccion supera el contexto del modelo configurado, para corregir la seleccion antes de alimentar un contexto que no va a caber.

### F4 -- Formatos de salida adicionales (P3)

**HU-10: Elegir el formato de salida del feed**
Como desarrollador cuyo destino no es un modelo que parsea XML, quiero elegir entre XML, Markdown y texto plano como formato de salida del feed, para que el contexto salga en el formato que mi destino espera.

**HU-11: Que el formato elegido se aplique a clipboard y export**
Como desarrollador que usa tanto el portapapeles como la exportacion a fichero, quiero que el formato de salida elegido se aplique a ambos por igual, para tener una salida consistente independientemente del canal.

## 6. Casos borde y decisiones de producto

Decididos en modo autopilot (sin usuario disponible). Cada decision se documenta con su porque.

### F1 -- SecretScanner

- **El aviso AVISA, no bloquea. Decision: avisa con opcion de continuar; default = Cancelar.**
  *Por que:* Pasture es una herramienta de desarrollador, y un escaner de patrones propios (sin dependencias) tendra falsos positivos (un `sk-` en un ejemplo de docs, una clave de prueba). Bloquear de forma dura convertiria la feature en un estorbo y empujaria al usuario a desactivarla. El patron correcto es el de "fail-safe with override": friccion solo cuando hay deteccion, default seguro (Cancelar), y escape explicito para el falso positivo. Esto preserva la security invariant sin sacrificar usabilidad.

- **Sin redactado automatico. Decision: si el usuario continua, se envia el contenido original.**
  *Por que:* Redactar (sustituir el secreto por `***`) cambiaria silenciosamente el contexto que el usuario cree estar alimentando, lo que es peor que avisar. Y abre la puerta a redactar de mas (romper contexto util). YAGNI para v1.4.

- **El escaneo cubre los tres canales: clipboard, export y Ask.**
  *Por que:* El canal mas peligroso es Ask (envio irreversible a un tercero), pero el portapapeles y un fichero exportado tambien son fugas. El hook va en `FeedService`/punto comun para cubrir los tres con una sola implementacion.

- **Catalogo de patrones acotado (v1.4):** `sk-ant-` (Anthropic), `ghp_`/`gho_`/`ghu_`/`ghs_` (GitHub tokens), claves AWS (`AKIA[0-9A-Z]{16}`), bloques PEM (`-----BEGIN ... PRIVATE KEY-----`), y `sk-` generico (OpenAI-style).
  *Por que:* Cubre los secretos mas comunes en el flujo de un dev que usa Claude Code. Ampliar el catalogo es trivial y aditivo en versiones futuras; no justifica retrasar v1.4.

### F2 -- Presets

- **Preset con ficheros que ya no existen. Decision: aplicar los que existen, avisar de los que faltan, NO borrar el preset.**
  *Por que:* Un preset es una conveniencia, no una fuente de verdad. Fallar en bloque porque falta 1 de 5 ficheros frustra al usuario. Borrar el preset automaticamente destruye trabajo del usuario por un cambio que puede ser temporal (fichero movido, renombrado fuera). La opcion equilibrada: degradar con gracia y dar visibilidad (toast con el conteo de ausentes). El usuario decide si actualiza el preset.

- **Referencia por path relativo a `~/.pasture/`, no por contenido ni por URL absoluta.**
  *Por que:* `~/.pasture/` es la fuente de verdad y los `MDFile` son `Identifiable` por URL. El path relativo sobrevive a que la carpeta se mueva de ubicacion (raro pero posible) y es estable frente a ediciones de contenido. No se guarda el contenido en el preset: eso seria duplicar estado y desincronizarse.

- **Selecciones independientes ventana/menu bar.** Aplicar un preset afecta solo a la superficie desde la que se aplica.
  *Por que:* Es el comportamiento ya establecido en v1.3.0 (las selecciones son independientes). Un preset no debe cambiar ese contrato.

- **Nombre duplicado. Decision: confirmar sobrescritura.**
  *Por que:* Evita perder un preset por un nombre repetido accidental, sin imponer nombres unicos rigidos.

### F3 -- Aviso de limite de contexto

- **El denominador solo aparece si hay modelo de AI configurado.** Sin modelo, se mantiene `~Xk tokens` a secas (comportamiento actual).
  *Por que:* El contexto (`contextWindow`) depende del modelo. Sin modelo no hay denominador con sentido. No introducir regresion en usuarios que solo usan Feed sin Ask.

- **El aviso salta al EXCEDER, no al acercarse.** Sin umbral intermedio (ej: 90%).
  *Por que:* Un umbral de "te estas acercando" es subjetivo y ruidoso. El estimador de tokens ya es heuristico (~4 chars/token); anadir un umbral difuso sobre una estimacion difusa da falsos avisos. Binario (cabe / no cabe) es mas honesto. YAGNI para el umbral intermedio.

- **El aviso es visual (color/icono), no un bloqueo del Feed.**
  *Por que:* Coherente con F1: Pasture avisa, el usuario decide. Ademas el conteo es una estimacion; bloquear sobre una estimacion seria incorrecto.

### F4 -- Formatos de salida

- **El formato aplica a clipboard, export Y al contexto de Ask.**
  *Por que:* El formato es una propiedad del *feed*, no del canal. Aplicarlo solo a unos canales crearia inconsistencia ("por que el portapapeles sale distinto al fichero?"). En Ask, los modelos modernos parsean Markdown y XML igual de bien, asi que respetar la preferencia del usuario es seguro y coherente.

- **Colision de fences en formato Markdown. Decision: ajustar la longitud del fence envolvente.**
  *Por que:* Si un fichero contiene ` ``` `, un bloque envolvente de 3 backticks se rompe. La solucion estandar (CommonMark) es usar un fence mas largo (4+ backticks) que el mas largo presente en el contenido. Esto preserva la fidelidad del contenido sin escapar caracter a caracter.

- **El formato XML por defecto se mantiene.** v1.4 no cambia el default; anade opciones.
  *Por que:* Retrocompatibilidad. Un usuario que actualiza no debe ver su feed cambiar de formato sin haberlo pedido.

## 8. Fuera de alcance (explicito)

- **Punto 5: conteo de tokens real via API.** El estimador heuristico (~4 chars/token) se mantiene. Llamar a un endpoint de tokenizacion (o portar un tokenizer) anade latencia, posible coste y/o una dependencia. NO en v1.4.
- **Servidor MCP / integracion MCP.** Fuera del alcance de "quick wins". Es una linea de producto distinta.
- **Redactado o enmascarado automatico de secretos.** F1 avisa; no modifica el contenido. (Ver decision en casos borde.)
- **Escaneo de secretos en tiempo real mientras se edita.** Pasture no edita inline (decision de v1.2). El escaneo ocurre en el momento del feed, no de forma continua.
- **Catalogo de secretos configurable por el usuario / patrones custom.** El catalogo de v1.4 es fijo. Anadir patrones es trabajo de codigo, no de configuracion de usuario, en esta version.
- **Presets que capturen ademas el modelo, el formato de salida o el destino de export.** Un preset en v1.4 es solo una lista de ficheros. Capturar mas estado es feature creep; se reconsidera si hay demanda.
- **Sincronizacion / export-import de presets entre maquinas.** Persistencia local en UserDefaults. Nada de iCloud ni ficheros de preset exportables.
- **Umbral de aviso "te acercas al limite" (ej: 90%).** El aviso es binario: cabe o no cabe. (Ver casos borde.)
- **Formatos de salida adicionales mas alla de los tres (ej: JSON, YAML, HTML).** Tres formatos cubren los destinos reales del usuario tipo. Mas formatos es feature creep.
- **"Todo en uno".** Recordatorio del Detective: `prompt_builder` murio asi. v1.4 = estos 4 puntos. Nada mas.

## 9. Riesgos y dependencias

| Riesgo | Impacto | Mitigacion |
|--------|---------|------------|
| Falsos positivos del SecretScanner generan friccion y empujan a "ignorar siempre" | Medio | Default Cancelar + override explicito por feed (no un "desactivar para siempre" facil). Catalogo de patrones especifico, no agresivo. Tests sobre un set de positivos y negativos conocidos. |
| Falsos negativos (un secreto no detectado pasa el filtro) | Alto (es seguridad) | Documentar que el escaner es una *red de seguridad best-effort*, no una garantia. Catalogo cubre las familias mas comunes del flujo dev. No se vende como infalible. |

| El nuevo formato Markdown rompe el contenido por colision de fences | Medio | Fence dinamico (longitud > maximo presente en el contenido), patron CommonMark. Tests con contenido que incluye backticks. |
| Regresion en el formato XML al refactorizar `ContextBuilder` para multi-formato | Alto | El formato XML debe producir salida byte-identica a v1.3.0. Test de snapshot. |
| El estimador de tokens (~4 chars/token) da una falsa sensacion de precision en el aviso de limite | Bajo | El indicador ya usa `~` (aproximado). El aviso es binario y honesto sobre su naturaleza heuristica. |

## 10. Dependencias tecnicas

- **Ninguna dependencia externa nueva.** Todo con frameworks de Apple. (Restriccion dura del proyecto.)
- **macOS 14+ (Sonoma)** sigue siendo el target minimo.
- **PastureKit** alberga la logica nueva testeable: `SecretScanner`, la generacion multi-formato (extension de `ContextBuilder`), el modelo de presets y el calculo de limite.
- **`FeedService`** es el punto de integracion para F1 (escaneo) y F2 (aplicar preset).
- **`ExportSettings` / patron de Settings** es la base para el picker de formato (F4).
- **`AISettings` (`model.contextWindow`)** es la fuente del denominador para F3.

## 11. Orden de implementacion sugerido

1. **F1 -- SecretScanner (P0).** Mayor valor de seguridad, 100% testeable en PastureKit antes de tocar UI. Se prueba el catalogo aislado, luego se engancha en `FeedService`.
2. **F4 -- Formatos de salida (P3, pero adelantado).** Se hace pronto porque toca `ContextBuilder`, el mismo nucleo que F1 escanea. Refactorizar el generador una sola vez evita retrabajo.
3. **F3 -- Aviso de limite (P2).** Casi solo UI, bajo riesgo. El dato ya existe.
4. **F2 -- Presets (P1).** Mayor superficie de UI (ventana + menu bar + gestion). Se deja para el final por ser el de mas piezas, aunque sea P1 en valor.

> Nota: la prioridad de *valor* (P0..P3) y el orden de *implementacion* no coinciden a proposito. F2 es P1 en valor pero se implementa al final por coste/superficie; F4 es P3 en valor pero se adelanta para no tocar `ContextBuilder` dos veces. El architect puede reordenar con justificacion.
