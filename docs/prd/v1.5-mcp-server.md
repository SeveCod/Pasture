# PRD: v1.5 -- Servidor MCP `pasture-mcp`

**Producto:** Pasture (macOS)
**Version objetivo:** 1.5.0
**Fecha:** 2026-06-12
**Autor:** El Buscador de Problemas (PO)
**Estado:** BORRADOR -- pendiente de aprobacion
**Base tecnica (no reabrir):** `docs/spikes/spike-mcp-server.md` (PoC validado, ADR-001/002/003 propuestos)

---

## 1. Problema

El usuario tipo de Pasture es un desarrollador que usa Claude Code y/o Claude Desktop a diario y que ya ha hecho el trabajo de **curar** su contexto en `~/.pasture/`: ficheros Markdown organizados en colecciones, depurados, listos para alimentar a un AI. Pasture v1.4 ya hace bien ese trabajo de curacion.

Pero hay un agujero en el ultimo metro del flujo. Hoy, para que ese contexto curado llegue al AI, el usuario tiene que **sacarlo de Pasture y meterlo a mano** en la conversacion:

- Abre Pasture (ventana o menu bar), selecciona los ficheros, hace Feed a clipboard, vuelve a Claude, pega.
- O exporta a un fichero y lo adjunta.

Es copy-paste manual, repetido en cada sesion, en cada nueva conversacion. El contexto vive en un sitio (`~/.pasture/`) y el AI vive en otro (Claude Code/Desktop), y el puente entre ambos es el portapapeles y los dedos del usuario.

**Dolor concreto:** El usuario que trabaja con Claude Code en su terminal no puede pedirle a Claude "lee mi contexto de la coleccion proyecto-X" y que Claude lo tenga, sin mas. Tiene que cambiar de ventana, seleccionar, copiar, pegar -- romper el flujo -- cada vez. El contexto curado esta a un copy-paste de distancia, pero ese copy-paste se paga una y otra vez.

Hay un agujero adicional, estrategico, no solo de fluidez:

**Pasture vive aislada del ecosistema de herramientas del usuario.** Claude Code se conecta a docenas de fuentes via MCP (Model Context Protocol): repos, bases de datos, APIs, otros vaults. Pasture, que es precisamente un almacen de contexto curado, no esta en esa red. El AI no sabe que `~/.pasture/` existe. Para el ecosistema MCP del usuario, el trabajo de curacion de Pasture es invisible.

---

## 2. Contexto

### Por que ahora

- Pasture v1.4.0 esta recien entregada (420 tests). El nucleo de curacion (`FileLibrary`, `PathValidator`, `ContextBuilder`, `MDFile`) es estable, publico y testeado -- justo las piezas que un servidor MCP necesita reutilizar.
- MCP se ha consolidado como el estandar de facto para conectar fuentes de contexto a clientes AI. El subset que necesita un servidor de solo lectura es pequeno y estable (version spec `2025-06-18`).
- **El spike ya esta cerrado y la viabilidad probada.** El PoC (`/tmp/pasture-mcp-spike`, 143 LOC, cero dependencias, Swift 6) demostro el handshake completo, las tools contra el vault real (68 ficheros `.md`), el rechazo de path traversal y latencias <10 ms. No partimos de cero ni de una hipotesis: partimos de un PoC funcionando.

### Tesis de producto

> Convertir Pasture de **"app de portapapeles"** en **hub de contexto** legible por el ecosistema MCP del usuario.

El patron "vault expuesto por MCP" es el **dominante** del espacio: `mcp-obsidian` (3.9k estrellas), `basic-memory` (3.2k estrellas). Y -- dato del Detective -- **ningun competidor nativo macOS lo tiene**. Pasture ya tiene el activo diferencial (contexto curado en colecciones + el formato Feed con `ContextBuilder`); lo que le falta es la puerta para que el AI entre a leerlo sin copy-paste. Esta feature abre esa puerta y, de paso, ocupa un hueco vacio en el mercado.

### Datos que respaldan la apuesta

- El flujo copy-paste es la friccion del "ultimo metro" que la propia razon de ser de Pasture (curar contexto para AI) deberia haber eliminado y no ha eliminado.
- La estrella de la feature no es exponer ficheros crudos (eso lo da cualquier server-filesystem generico): es la tool `feed_context`, que entrega el contexto **ensamblado con el formato Feed de Pasture** (`ContextBuilder`, XML+CDATA o el formato configurado). Eso es lo que un server generico no puede dar y lo que hace que Pasture no sea "una carpeta mas".

## 3. Solucion propuesta (alto nivel)

> El marco tecnico esta fijado por el spike y sus tres ADRs (Propuestos). Este PRD vive **dentro** de ese marco; no lo reabre. El detalle de implementacion es del architect y del senior-dev.

Un **cuarto target SPM executable** (`pasture-mcp`), Swift puro, **cero dependencias**, que habla **JSON-RPC sobre stdio** y expone `~/.pasture/` **en solo lectura** a clientes MCP (Claude Code, Claude Desktop).

Marco heredado del spike (no se decide aqui, se respeta):

- **ADR-001:** implementacion propia, sin SDK. (~6 metodos JSON-RPC, transporte delimitado por newline.)
- **ADR-002:** binario embebido en `Pasture.app/Contents/MacOS/pasture-mcp`. Riesgo Gatekeeper **ya despejado empiricamente**: el exec por un cliente MCP no dispara el assessment; conecta incluso con quarantine.
- **ADR-003:** solo lectura, raiz unica `~/.pasture/`.

Tools expuestas (cinco):

| Tool | Responsabilidad | Reutiliza |
|------|-----------------|-----------|
| `list_files` | Listar ficheros y colecciones del vault. | `FileLibrary` |
| `read_file` | Devolver el contenido crudo de un fichero. | `PathValidator` + I/O |
| `search` | Buscar texto en el vault, devolver ficheros que casan. | `MDFile.matches` |
| `feed_context` | **Tool estrella.** Devolver contexto **ensamblado** con el formato Feed de Pasture, a partir de una coleccion o de una lista de ficheros. | `ContextBuilder` |

Y en la app: la **UX de registro** -- un generador de configuracion ("Copiar configuracion MCP") que produce el snippet listo para `claude mcp add` y para `claude_desktop_config.json`, con la **ruta real del binario** dentro del bundle instalado.

> Nota de alcance: el spike menciona `get_context(collection)` y `feed_context` como dos tools. **Decision de producto (ver seccion 6):** se fusionan en una sola, `feed_context`, parametrizada para aceptar coleccion o lista de ficheros. Una tool, no dos que hacen casi lo mismo. YAGNI.

## 4. Historias de usuario priorizadas

Prioridad por relacion valor/riesgo. La UX de instalacion (descubrir y registrar el server desde la app) es el **make-or-break**: un servidor MCP que el usuario no consigue registrar no existe. Por eso encabeza la lista pese a no ser "la feature".

| # | Epica | Prioridad | Justificacion |
|---|-------|-----------|---------------|
| P0 | Registro del server desde la app (UX de instalacion) | **Critica** | Make-or-break. Sin registro sin friccion, nada del resto se usa. |
| P0 | Handshake y ciclo de vida MCP | **Critica** | Sin protocolo correcto, el cliente no conecta. Base de todo. |
| P1 | `feed_context` (tool estrella) | **Alta** | Es el valor diferencial: contexto ensamblado, no ficheros crudos. La razon de construir esto. |
| P1 | `list_files` (ficheros + colecciones) | **Alta** | El AI necesita descubrir que hay en el vault antes de pedir nada. |
| P2 | `read_file` | **Media** | Util, pero es lo que cualquier server-filesystem ya da. Valor incremental. |
| P2 | `search` | **Media** | Util para vaults grandes; el AI localiza antes de leer. |
| P3 | Comportamiento ante errores | **Transversal** | No es una epica aparte: cada tool debe degradar con gracia. Se especifica como criterios. |

### Epica A -- Registro del server desde la app (P0, make-or-break)

**HU-1: Descubrir que Pasture puede actuar como servidor MCP**
Como desarrollador que ya usa Pasture y Claude Code, quiero ver en la app que existe una forma de exponer mi vault a Claude, para enterarme de que esta capacidad existe sin tener que leer documentacion externa.

**HU-2: Copiar la configuracion MCP lista para Claude Code**
Como desarrollador que usa Claude Code, quiero copiar de un clic el comando `claude mcp add` con la ruta real del binario ya rellenada, para registrar el server sin construir el comando a mano ni averiguar donde esta el ejecutable.

**HU-3: Copiar la configuracion MCP lista para Claude Desktop**
Como desarrollador que usa Claude Desktop, quiero copiar el bloque JSON de `mcpServers` con la ruta real del binario, para pegarlo directamente en `claude_desktop_config.json` sin editar rutas a mano.

### Epica B -- Handshake y ciclo de vida (P0)

**HU-4: Que el cliente MCP conecte con el server**
Como desarrollador que ha registrado el server, quiero que Claude Code/Desktop muestre el server como conectado, para confirmar que el puente entre mi vault y el AI esta operativo.

**HU-5: Que el server liste sus tools al cliente**
Como cliente MCP (Claude), quiero recibir el catalogo de tools con su esquema de entrada, para saber que puedo pedirle al vault de Pasture.

### Epica C -- `feed_context` (tool estrella, P1)

**HU-6: Obtener el contexto ensamblado de una coleccion**
Como desarrollador que trabaja en un proyecto, quiero pedirle a Claude "trae el contexto de la coleccion proyecto-X" y que reciba todos los ficheros de esa coleccion **ensamblados con el formato Feed de Pasture**, para alimentar mi contexto curado sin copy-paste.

**HU-7: Obtener el contexto ensamblado de una lista de ficheros**
Como desarrollador que quiere un subconjunto concreto, quiero pedir el contexto ensamblado de una seleccion de ficheros por nombre, para componer exactamente el contexto que necesito sin traer la coleccion entera.

### Epica D -- `list_files` (P1)

**HU-8: Listar los ficheros y colecciones del vault**
Como cliente MCP (Claude), quiero listar los ficheros del vault agrupados o etiquetados por coleccion, para que el AI sepa que contexto hay disponible antes de pedirlo.

### Epica E -- `read_file` (P2)

**HU-9: Leer el contenido de un fichero concreto**
Como desarrollador que quiere un unico fichero crudo (sin envoltorio Feed), quiero pedir su contenido por ruta, para inspeccionar un fichero puntual del vault.

### Epica F -- `search` (P2)

**HU-10: Buscar texto en el vault**
Como desarrollador con un vault de decenas/cientos de ficheros, quiero que Claude busque que ficheros mencionan un termino, para localizar el contexto relevante sin enumerar todo el vault.

## 5. Criterios de aceptacion

Formato Given/When/Then. Listos para test. Los identificadores de tools y campos van en ingles.

### HU-1: Descubrir la capacidad MCP

```
Given Pasture v1.5 instalada y el usuario en la ventana de Settings/preferencias
When abre la seccion correspondiente (p.ej. una pestana "MCP")
Then ve una explicacion breve de que Pasture puede exponer ~/.pasture/ en solo lectura a clientes MCP
  And ve los controles para copiar la configuracion (HU-2, HU-3)
```

### HU-2: Copiar config para Claude Code

```
Given Pasture instalada en /Applications/Pasture.app
When el usuario pulsa "Copiar configuracion (Claude Code)"
Then el portapapeles contiene un comando "claude mcp add ... -- <ruta> " donde <ruta> es la ruta absoluta real al binario embebido en el bundle instalado
  And la ruta apunta a Contents/MacOS/pasture-mcp dentro del .app desde el que se ejecuta la app
```

```
Given la app se ejecuta desde una ubicacion distinta de /Applications (p.ej. ~/Downloads)
When el usuario copia la configuracion
Then la ruta del binario refleja la ubicacion real actual del .app, no una ruta hardcodeada
```

### HU-3: Copiar config para Claude Desktop

```
Given Pasture instalada
When el usuario pulsa "Copiar configuracion (Claude Desktop)"
Then el portapapeles contiene un fragmento JSON valido con una entrada en "mcpServers" (clave del server, "command" con la ruta real del binario, "args" si procede)
  And el JSON es pegable directamente dentro de claude_desktop_config.json sin edicion de rutas
```

### HU-4: El cliente conecta

```
Given el usuario ha registrado pasture-mcp con la configuracion copiada de la app
When el cliente MCP arranca el server e inicia el handshake (initialize)
Then el server responde con protocolVersion soportada, capabilities { tools: {} } y serverInfo
  And tras notifications/initialized el cliente muestra el server como conectado
```

```
Given el cliente envia un mensaje "ping"
When el server lo recibe
Then responde {} (mantiene la conexion viva)
```

```
Given el cliente cierra stdin (EOF)
When el server detecta el cierre
Then el proceso termina limpiamente sin requerir SIGKILL
```

### HU-5: Listar tools

```
Given el handshake completado
When el cliente envia "tools/list"
Then el server devuelve el catalogo con list_files, read_file, search y feed_context
  And cada tool incluye un inputSchema con type "object"
```

### HU-6: feed_context de una coleccion

```
Given una coleccion "proyecto-X" con 3 ficheros .md en ~/.pasture/proyecto-X/
When el cliente llama feed_context con el parametro de coleccion = "proyecto-X"
Then el server devuelve el contexto de los 3 ficheros ensamblado con ContextBuilder (formato Feed de Pasture, XML+CDATA o el formato Feed configurado)
  And el resultado es identico en estructura al Feed que produciria la app para esa misma seleccion
```

```
Given una coleccion con un fichero cuyo contenido incluye la secuencia de cierre CDATA "]]>"
When el cliente llama feed_context sobre esa coleccion
Then la secuencia se escapa correctamente (sin romper el envoltorio, sin inyeccion), igual que en el Feed de la app
```

```
Given el parametro de coleccion referencia una coleccion que no existe
When el cliente llama feed_context
Then el server responde con isError: true y un mensaje claro ("coleccion no encontrada"), sin tumbar la conexion
```

### HU-7: feed_context de una lista de ficheros

```
Given los ficheros "a.md" y "notas/b.md" existen en el vault
When el cliente llama feed_context con una lista de ficheros = ["a.md", "notas/b.md"]
Then el server devuelve el contexto de ambos ensamblado con ContextBuilder, en el orden pedido
```

```
Given una lista de ficheros que incluye uno inexistente ("a.md", "fantasma.md")
When el cliente llama feed_context
Then el server ensambla el contexto de los ficheros que existen
  And el resultado indica (en el contenido o como aviso) que "fantasma.md" no se encontro
  And NO falla en bloque por un fichero ausente
```

```
Given una llamada a feed_context sin coleccion ni lista de ficheros (argumentos vacios)
When el cliente la invoca
Then el server responde con isError: true y un mensaje que explica que se requiere coleccion o lista de ficheros
```

### HU-8: Listar ficheros y colecciones

```
Given un vault con 5 ficheros en la raiz y 2 colecciones ("proyecto-X" con 3, "notas" con 4)
When el cliente llama list_files
Then el server devuelve los 12 ficheros, cada uno identificado por su ruta relativa a ~/.pasture/ y con su coleccion (o raiz) indicada
  And los ficheros ocultos y los symlinks quedan excluidos (igual que FileLibrary en la app)
```

```
Given un vault vacio (~/.pasture/ existe pero no tiene ficheros .md)
When el cliente llama list_files
Then el server devuelve una lista vacia con exito (no es un error: un vault vacio es un estado valido)
```

### HU-9: Leer un fichero

```
Given el fichero "proyecto-X/diseno.md" existe en el vault
When el cliente llama read_file con esa ruta
Then el server valida la ruta con PathValidator (dentro de ~/.pasture/) y devuelve su contenido crudo
```

```
Given el cliente llama read_file con "../.zshrc" o "../../../../etc/passwd"
When el server valida la ruta
Then PathValidator la rechaza y el server responde isError: true ("ruta fuera del vault"), sin leer nada fuera de ~/.pasture/
```

```
Given el cliente llama read_file con una ruta inexistente dentro del vault ("no-existe.md")
When el server intenta leer
Then responde isError: true con un mensaje claro ("fichero no encontrado"), sin tumbar la conexion
```

### HU-10: Buscar

```
Given un vault donde 2 ficheros contienen el termino "deployment"
When el cliente llama search con query = "deployment"
Then el server devuelve los 2 ficheros que casan (busqueda literal, case-insensitive, sobre nombre y contenido, igual que MDFile.matches)
```

```
Given una query vacia ("")
When el cliente llama search
Then el server devuelve lista vacia (o el conjunto que defina MDFile.matches para query vacia), con exito, sin recorrer/volcar todo el vault como "match"
```

```
Given una query que un motor de regex interpretaria como patron costoso (p.ej. "(a+)+$")
When el cliente llama search
Then el server la trata como texto literal (no como regex), sin riesgo de ReDoS ni de lectura masiva descontrolada
```

## 6. Casos borde y decisiones de producto

Decididas en modo autopilot (autopilot del PO, sin reabrir el spike). Cada una con su porque. **Estas son las decisiones de producto que me tocan a mi**, distintas de las decisiones tecnicas ya tomadas en los ADRs.

### D1 -- `search`: literal o con sintaxis

**Decision: busqueda LITERAL, case-insensitive, sin sintaxis de query (sin regex, sin operadores, sin comodines).**

*Por que:* (a) Es exactamente lo que ya hace `MDFile.matches(query:)` en la app -- reutilizamos el predicado existente, una sola semantica de busqueda en todo el producto, cero divergencia entre la app y el server. (b) El spike marca explicitamente el riesgo R5 (ReDoS / lectura masiva si `search` no esta acotado); la busqueda literal lo elimina por diseno, no por mitigacion. (c) Una sintaxis de query es una funcionalidad que el usuario no ha pedido y que el AI cliente no necesita: el modelo refina por iteracion (busca, lee, vuelve a buscar), no por operadores booleanos. YAGNI. Si en el futuro hace falta, es aditivo.

### D2 -- `feed_context`: coleccion, lista de ficheros, o ambos

**Decision: AMBOS, en una sola tool `feed_context` con dos parametros mutuamente complementarios (coleccion O lista de ficheros). Se fusionan las dos tools que el spike listaba por separado (`get_context(collection)` y `feed_context`).**

*Por que:* (a) Los dos son el mismo verbo -- "ensamblame contexto con el formato Feed" -- y solo cambian en el selector de entrada. Dos tools para un verbo es ruido para el modelo cliente, que tiene que decidir cual usar. Una tool con un parametro de seleccion es mas simple de descubrir y de razonar. (b) Cubre los dos casos de uso reales: el de grano grueso (coleccion entera, el caso dominante del usuario que organiza por proyecto) y el de grano fino (subconjunto concreto). (c) Si se pasan los dos, gana la lista de ficheros (mas especifica) o se devuelve error de argumentos -- el architect lo concreta; el contrato de producto es "uno de los dos, no ninguno" (ver criterio de argumentos vacios en HU-7). YAGNI sobre una tercera tool: no hay `get_context` y `feed_context` separadas.

### D3 -- Templates de Pasture en `feed_context`: renderizados o crudos

**Decision: se sirven CRUDOS, es decir, con las plantillas SIN renderizar (los `{{VAR}}` se entregan tal cual, sin sustituir). Default sensato y unico en v1.5: sin renderizado.**

*Por que:* (a) Renderizar una plantilla exige **valores de variables**, y en un protocolo stdio sin UI no hay forma de pedirselos al usuario en el momento. Las unicas opciones serian: renderizar con los defaults del template (`{{VAR=default}}`), o servir crudo. (b) Servir crudo es lo mas honesto y lo menos sorprendente: el cliente AI recibe el fichero tal como esta en el vault, con sus marcadores visibles. Un modelo moderno entiende perfectamente un `{{PLACEHOLDER}}` como "aqui va X"; de hecho, le da informacion (sabe que ese hueco es parametrizable). (c) Renderizar con defaults silenciosamente inyectaria valores que el usuario quiza no quiere, sin que el usuario pueda intervenir -- el mismo antipatron que rechazamos en el SecretScanner de v1.4 (no modificar el contexto que el usuario cree estar alimentando). (d) El renderizado de templates es una operacion interactiva por naturaleza (la app pide los valores en una hoja antes del Feed); trasplantarla a un canal sin UI seria forzarla. **Coherencia con la app:** en la app, el Feed sin rellenar variables tambien deja los `{{VAR}}` crudos hasta que el usuario los completa. El server hereda ese mismo estado "sin completar". YAGNI: el renderizado parametrizado por MCP queda fuera de alcance (seccion 8).

### D4 -- SecretScanner (v1.4) en el canal MCP

**Decision: el SecretScanner NO actua en el canal MCP en v1.5. En su lugar, se mitiga con (a) documentacion explicita de que el vault no es un almacen de secretos y (b) la propia naturaleza del registro como punto de consentimiento.**

*Por que:* Este es el caso borde mas delicado, y lo razono entero.

- **El SecretScanner de v1.4 es interactivo por diseno: avisa y el usuario decide (default Cancelar).** Ese patron "avisa, el usuario decide" **no es trasladable a un protocolo sin UI.** Un cliente MCP que llama `read_file` o `feed_context` no puede recibir un dialogo "se ha detectado un posible secreto, continuar?" -- no hay quien lo conteste en el momento, no hay usuario en el loop de esa llamada. La premisa misma del SecretScanner (un humano frente a una decision) no existe aqui.

- **Las alternativas sin UI son peores:**
  - *Bloquear* la lectura de cualquier fichero con un patron de secreto convertiria al server en impredecible: el AI pediria un fichero y recibiria un error opaco por un falso positivo (un `sk-` en un ejemplo de docs). Rompe la utilidad sin ganar seguridad real -- el usuario ya consintio exponer el vault al registrarlo.
  - *Redactar* silenciosamente (sustituir el secreto por `***`) cambiaria el contenido que el usuario cree tener en el vault, **exactamente el antipatron que ya rechazamos en v1.4** ("no modificar el contexto que el usuario cree estar alimentando"). Y un redactado con falsos positivos romperia contexto util sin que nadie pueda intervenir.

- **El modelo de confianza correcto es el del consentimiento en el registro, no el del aviso por llamada.** El spike (seccion 5) ya lo establece: en stdio, *quien puede lanzar el binario ya tiene acceso al FS del usuario*; el acto de **registrar** el server (HU-2/HU-3, un paso explicito y deliberado del usuario) **es** el punto de consentimiento. El usuario que registra `pasture-mcp` esta diciendo "expongo mi vault curado a mi cliente AI". Eso es una decision consciente, hecha una vez, en un punto con UI (la app) -- no docenas de decisiones por llamada en un canal sin UI.

- **`~/.pasture/` no es, por diseno, un almacen de secretos.** Es contexto **compartible** y curado. El riesgo residual (el usuario mete un `.env` en el vault) es el mismo que ya existe hoy con el Feed manual, y la mitigacion correcta es **documentar** que el vault es contexto compartible, no un baul de credenciales -- tal como recomienda el spike (riesgo residual no tecnico, seccion 5).

- **Coherencia v1.4 -> v1.5:** el SecretScanner sigue protegiendo los canales **interactivos** (clipboard, export, Ask) donde el patron "avisa, el usuario decide" si tiene sentido. El canal MCP es **no interactivo y consentido en el registro**: su control de seguridad es el consentimiento + `PathValidator` + filtrado de symlinks + raiz unica, no un scanner que no puede preguntar a nadie.

> Resumen D4: avisar no es posible en un protocolo sin UI; bloquear o redactar es peor que el problema; el consentimiento vive en el acto de registrar el server; el control de seguridad real es la raiz unica solo-lectura + PathValidator. El SecretScanner queda fuera de este canal, y se documenta que el vault no es para secretos.

### D5 -- Paginacion de `list_files`

**Decision: SIN paginacion en v1.5. `list_files` devuelve el vault completo en una sola respuesta.**

*Por que:* (a) El vault tipico del usuario tipo son **decenas a cientos** de ficheros `.md` (el PoC corrio contra 68). Una lista de ese tamano es trivial: nombres + colecciones, unos pocos KB, muy por debajo de cualquier limite practico de una respuesta MCP. (b) La paginacion (cursores, estado entre llamadas) anade complejidad de protocolo y de cliente para resolver un problema que el vault tipico no tiene. Introducirla ahora es optimizacion prematura. YAGNI. (c) Si algun usuario llega a tener miles de ficheros, la paginacion es aditiva y no rompe el contrato (se anade un cursor opcional). El default correcto para v1.5 es la simplicidad. **Nota:** `list_files` devuelve metadatos (ruta + coleccion), no el contenido de los ficheros -- el peso es bajo aunque haya cientos.

### D6 -- Errores: convencion de canal

**Decision: los errores de TOOL (path fuera del vault, fichero/coleccion inexistente, argumentos invalidos) se devuelven como `isError: true` dentro de `result` -- visibles para el modelo, que puede reaccionar. Los errores de PROTOCOLO (JSON malformado, metodo desconocido, params invalidos a nivel JSON-RPC) usan el objeto `error` de JSON-RPC (`-32700`, `-32601`, `-32602`).**

*Por que:* Es la distincion correcta de MCP (gotcha 6 del spike) y la que permite que el AI se recupere: un `feed_context` a una coleccion inexistente no debe tumbar la conexion ni parecer un fallo del server, sino devolverle al modelo un "esa coleccion no existe" que el modelo puede leer y corregir (p.ej. llamando antes a `list_files`). Un vault vacio o una query vacia **no son errores**: son estados validos que devuelven resultado vacio con exito.

## 8. Fuera de alcance (explicito)

- **Escritura.** El server es solo lectura (ADR-003). Crear/editar/borrar ficheros del vault desde el cliente MCP queda fuera. Requeriria un ADR nuevo sobre locking entre procesos y la interaccion con el `DirectoryWatcher`.
- **Multiples roots.** Raiz unica `~/.pasture/` (ADR-003). Nada de exponer otras carpetas ni hacer la raiz configurable.
- **HTTP transport (Streamable HTTP / SSE).** Solo stdio. Nada de servidor de red, puertos ni conexiones entrantes.
- **Autenticacion.** El modelo de confianza de stdio es "quien lanza el proceso ya tiene acceso al FS"; el consentimiento vive en el registro. Anadir auth a stdio seria teatro de seguridad.
- **Notarizacion / firma de la GUI.** El riesgo Gatekeeper ya esta despejado empiricamente (el exec por un cliente MCP no dispara el assessment). v1.5 no aborda firmar ni notarizar el `.app`.
- **Renderizado de templates con valores via MCP.** Los templates se sirven crudos (D3). Un mecanismo para pasar valores de variables por el canal MCP queda fuera; es interactivo por naturaleza y no encaja en stdio sin UI.
- **SecretScanner en el canal MCP.** No actua en este canal (D4). Sigue protegiendo clipboard/export/Ask.
- **Sintaxis de query / regex en `search`.** Busqueda literal (D1). Operadores, comodines y regex fuera de alcance.
- **Paginacion / streaming de respuestas.** `list_files` y `feed_context` devuelven respuesta completa (D5).
- **Resources, prompts, logging, completions, sampling, elicitation MCP.** Subset solo-tools. La mitad compleja de la spec MCP queda fuera (heredado del spike).
- **"Todo en uno".** Recordatorio permanente del Detective (`prompt_builder` murio asi). v1.5 = un servidor MCP de solo lectura con cuatro tools y su UX de registro. Nada mas.

## 9. Riesgos y dependencias

| Riesgo | Impacto | Mitigacion |
|--------|---------|------------|
| La UX de registro es confusa y el usuario no consigue conectar el server (make-or-break) | **Alto** | Generador de config con ruta real ya rellenada (HU-2/HU-3); snippets distintos por cliente; texto explicativo. Es la epica P0. |
| `stdout` contaminado por un `print()` perdido rompe el protocolo | Alto | Invariante de implementacion: `stdout` solo mensajes MCP, todo log a `stderr` (gotcha 1-2 del spike). Responsabilidad del senior-dev; el PO lo marca como criterio de no-regresion. |
| El usuario guarda secretos en `~/.pasture/` y un cliente MCP los lee | Medio | Documentar que el vault es contexto compartible, no almacen de secretos (D4). Riesgo residual identico al del Feed manual de hoy. |
| `feed_context` diverge del formato Feed de la app | Medio | Reutilizar `ContextBuilder` (la misma pieza que usa la app). Test golden que compare la salida del server con la del Feed de la app. |
| Path traversal via parametro de ruta alucinado por el LLM cliente | Alto (seguridad) | Toda ruta por `PathValidator` antes de I/O; filtrado de symlinks (`FileLibrary`). Ya validado en el PoC. |
| Fichero a medio escribir por el editor externo se lee parcial | Bajo | Riesgo transitorio identico al que ya asume la app. No justifica locking (solo lectura). |
| Drift de spec MCP | Bajo | Subset estable (solo-tools/stdio, la parte que menos cambia); version negociada en `initialize`. |

### Dependencias

- **Ninguna dependencia externa nueva** (ADR-001, invariante del proyecto).
- **PastureKit** aporta la logica de dominio reutilizada: `FileLibrary`, `PathValidator`, `ContextBuilder`, `MDFile` (`matches`). El target MCP no reimplementa nada de esto.
- **`bundle.sh`** debe copiar el segundo ejecutable al `.app` y la app debe generar el snippet con la ruta del bundle (ADR-002).
- **macOS 14+ (Sonoma)** sigue siendo el target minimo.
- **Spike cerrado** (`docs/spikes/spike-mcp-server.md`) y sus ADR-001/002/003 (Propuestos) como marco tecnico. El security-officer firma el diseno de implementacion en su fase.
