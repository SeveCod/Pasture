## 0. Alcance y modelo de confianza

### Qué cambia en v1.4 (superficie nueva)

| Feature | Componente nuevo / tocado | Riesgo de seguridad |
|---------|---------------------------|---------------------|
| F1 -- SecretScanner | `SecretScanner` (PastureKit) + hooks en `FeedService` y ruta de Ask | **Alto** -- es seguridad, e introduce el primer motor regex del proyecto |
| F2 -- Presets | Modelo `SelectionPreset` (Codable) + persistencia UserDefaults | Medio -- persiste rutas de fichero; entrada de usuario en nombres |
| F3 -- Aviso de límite | Extensión de `selectionSummary` (SidebarView) | Bajo -- solo UI, reutiliza `TokenEstimator` |
| F4 -- Formatos de salida | Extensión de `ContextBuilder` (XML/Markdown/plano) | **Alto** -- inyección por formato; el escaping CDATA actual NO se traslada solo |

### Modelo de confianza (lo que NO cambia)

- **Frontera de confianza principal:** la máquina local del usuario es de confianza. El único canal de salida hacia un tercero es la API de Anthropic/OpenRouter en Ask (HTTPS, ATS por defecto). Clipboard y export son superficies de fuga *local* pero bajo control del usuario.
- **El atacante relevante NO es un proceso remoto.** Es:
  1. El propio usuario filtrando un secreto sin querer (el problema que F1 ataca).
  2. Contenido hostil dentro de un fichero `.md` que el usuario importó (un `.md` con payloads diseñados para romper el formato de salida o el escáner) → relevante para F1 (ReDoS) y F4 (inyección de formato).
- **Confianza cero sobre el contenido de los ficheros.** Un `.md` puede venir de un import de PDF/CSV/DOCX, de un paste, o de un repo de terceros. **Todo contenido de fichero es hostil hasta que se demuestre lo contrario.** Esto ya está asumido en v1.3.0 (CDATA escaping) y debe mantenerse.

### Verificación de anclajes en el código (hecha, no asumida)

- `ContextBuilder.contextTag` (línea 26) **ya escapa** `]]>` → `]]]]><![CDATA[>`. Esta es la defensa anti-inyección CDATA existente. **F4 debe replicar una defensa equivalente en cada formato nuevo, no heredarla gratis.**
- `FeedService.deliverFeed` es el embudo de clipboard + export. **Pero Ask NO pasa por aquí**: `AskViewModel.send(context:contextTokens:)` recibe el contexto ya construido y llama a `AIClient.shared.ask()`. → **F1 necesita DOS puntos de enganche**, no uno.
- `Package.swift`: cero dependencias. Ningún `NSRegularExpression`/`Regex<>` en el código actual. F1 introduce el primer motor de regex → **vector ReDoS nuevo que antes no existía**.

## 1. Análisis STRIDE por feature

### F1 -- SecretScanner

| STRIDE | Amenaza | Severidad | Mitigación |
|--------|---------|-----------|------------|
| **Spoofing** | N/A (sin identidades nuevas) | -- | -- |
| **Tampering** | Contenido `.md` diseñado para evadir el escáner (falso negativo) o para colgarlo (ReDoS) | Alta | Patrones acotados y testeados (SEC-2); regex no-catastróficos + límite de tamaño/timeout (SEC-3) |
| **Repudiation** | Override del aviso sin rastro de qué se decidió | Baja | El override es por-feed y efímero, no persistente (SEC-6). No se requiere log de auditoría en app local mono-usuario. |
| **Information Disclosure** | **El secreto detectado acaba en logs, en la UI del aviso, o en un crash report** | **Alta** | El escáner reporta SOLO tipo de patrón + fichero + offset/línea. **NUNCA el valor del match** (SEC-4). |
| **Information Disclosure** | Falsa sensación de seguridad: el usuario cree que el escáner garantiza que no hay secretos, baja la guardia, y filtra un secreto no cubierto por el catálogo | **Alta** (es el riesgo central de esta feature) | El escáner es una red best-effort, NO una garantía. Comunicarlo en la UI (SEC-5). |
| **Denial of Service** | **ReDoS**: un fichero grande o con una cadena patológica hace que un regex con backtracking catastrófico cuelgue la app | **Alta** | Regex lineales/anclados + límite de tamaño de escaneo + escaneo fuera del main actor (SEC-3). |
| **Elevation of Privilege** | N/A | -- | -- |

**Notas F1:**

- **¿Debe loggearse el secreto?** No. Rotundamente no. Si el escáner detecta `sk-ant-api03-XXXX`, lo que puede aparecer en logs/UI/telemetría es: `"clave Anthropic detectada en api-keys.md, línea 12"`. **El valor `sk-ant-api03-XXXX` no se loguea, no se muestra en el diálogo, no se copia a ningún sitio.** "Si no está cifrado, no existe" -- y un secreto en un log de texto plano no está cifrado. Loguear el secreto que estás intentando proteger sería la ironía más cara del proyecto.
- **Falsos negativos = el riesgo número uno.** El catálogo (Anthropic, GitHub, AWS, PEM, OpenAI-style) cubre lo común, no lo todo. Una clave de Stripe, una de Google, un JWT, una cadena de conexión de base de datos con password → no se detectan en v1.4. Si la UI sugiere "no se detectaron secretos = es seguro enviar", estamos mintiendo. Debe decir "no se detectaron secretos *conocidos*".
- **Ruta de Ask:** la pregunta del usuario (`question`) también viaja a la API. Si el usuario pega un secreto en la caja de pregunta, el escáner de contexto no lo ve. Recomendación: escanear también la pregunta antes de enviar en Ask (SEC-1 lo cubre como "todo lo que sale").

### F2 -- Presets de selección

| STRIDE | Amenaza | Severidad | Mitigación |
|--------|---------|-----------|------------|
| **Tampering** | Nombre de preset con caracteres de control / payloads que rompan la UI o se interpreten al renderizar | Media | Sanitizar/validar el nombre del preset (SEC-8) |
| **Tampering** | Un preset apunta a una ruta fuera de `~/.pasture/` (path traversal vía `../`) si la referencia se resuelve sin validar | **Alta** | Resolver toda ruta de preset a través de `PathValidator.isInside` antes de cargar (SEC-9) |
| **Information Disclosure** | UserDefaults (plist en `~/Library/Preferences/`) queda con **rutas de ficheros** y nombres de proyecto del usuario en claro | Baja | Aceptable: son rutas *relativas* a `~/.pasture/` (nombres de fichero), no contenido ni secretos. Documentado (SEC-7). |
| **Denial of Service** | Número ilimitado de presets / nombres enormes inflan el plist | Baja | Límite razonable de nombre y conteo (SEC-8) |

**Notas F2:**

- **¿Qué se persiste en UserDefaults?** Según el PRD: nombre del preset + lista de rutas **relativas a `~/.pasture/`**. Esto es aceptable: una ruta relativa (`proyecto-x/contexto.md`) revela un nombre de fichero, no su contenido ni un secreto. El plist de Preferences es legible por cualquier proceso que corra como el usuario, pero ese ya es el modelo de la app (las API keys NO van aquí, van al Keychain -- mantener ese invariante: **un preset jamás guarda contenido de fichero ni nada sensible**).
- **Path traversal:** el peligro real. Si un preset guardara `../../../etc/passwd` como ruta relativa y al aplicarlo se resolviera contra `~/.pasture/` sin validar, se podría seleccionar/leer un fichero fuera del sandbox lógico. La invariante del proyecto (`isInsidePasture()` / `PathValidator.isInside`) **debe aplicarse al resolver un preset**, igual que se aplica a cualquier otra operación de fichero. No se exime a los presets de esta regla.
- **Nombres raros:** un nombre con `\0`, saltos de línea o caracteres de control podría romper la presentación o, peor, confundir al usuario sobre qué preset está aplicando (homoglyph/spoofing visual). Reutilizar la filosofía de `FilenameSanitizer` para los nombres de preset (no necesariamente el mismo set de reglas, pero sí validación).

### F4 -- Formatos de salida (XML / Markdown / plano)

| STRIDE | Amenaza | Severidad | Mitigación |
|--------|---------|-----------|------------|
| **Tampering** | **Inyección de formato Markdown**: un fichero con ` ``` ` rompe el bloque envolvente y el contenido posterior se interpreta como estructura, no como dato | **Alta** | Fence dinámico (longitud > máximo run de backticks presente) -- CommonMark (SEC-11) |
| **Tampering** | **Formato plano sin envoltura**: contenido con el separador elegido (p.ej. una línea `---` o `===`) se confunde con un límite de fichero → el consumidor mal-parsea dónde acaba un fichero y empieza otro | **Alta** | Separador inequívoco o que incluya el nombre del fichero; documentar que plano es lossy por diseño (SEC-12) |
| **Tampering** | **Regresión en XML**: refactorizar `ContextBuilder` para multi-formato rompe el escaping `]]>` existente | **Alta** | El formato XML debe producir salida byte-idéntica a v1.3.0. Test de snapshot (SEC-10) |
| **Tampering** | El nombre de fichero se inyecta en la cabecera Markdown (`## name`) o XML (`name="..."`) sin escapar | Media | XML ya escapa vía `xmlEscapedAttribute`; Markdown debe neutralizar saltos de línea / `#` en el nombre (SEC-11) |

**Notas F4 -- requisitos de escaping por formato (esto es el corazón de F4 desde seguridad):**

El principio: **cada formato de salida debe garantizar que el contenido de un fichero no puede romper la estructura envolvente ni hacerse pasar por estructura.** Es exactamente la garantía que `]]>` → `]]]]><![CDATA[>` da hoy en XML. Cada formato nuevo necesita su equivalente:

- **XML+CDATA (existente, default):** mantener intacto. Escape de `]]>` en contenido + `xmlEscapedAttribute` en el nombre. **Salida byte-idéntica a v1.3.0** (SEC-10).
- **Markdown fenced:**
  - El fence envolvente debe ser **más largo** que el run de backticks más largo presente en el contenido. Si el contenido tiene ` ```` ` (4 backticks), el envolvente usa 5+. Patrón CommonMark estándar. (SEC-11)
  - El nombre de fichero en `## name` debe neutralizar saltos de línea y no permitir que un `#` o un salto inyecte cabeceras falsas. (SEC-11)
  - **No** se debe usar HTML-escaping del contenido (rompería fidelidad); la defensa es el fence dinámico, no escapar el contenido carácter a carácter.
- **Texto plano:**
  - Sin envoltura significa que el separador entre ficheros es la **única** marca de estructura. Si el contenido puede contener el separador, el formato es ambiguo. Requisito: el separador debe incluir el **nombre del fichero** (p.ej. una línea cabecera tipo `=== name ===`) de forma que, aunque el contenido contenga la cadena separadora, el consumidor humano/máquina tenga una marca contextual. (SEC-12)
  - Documentar explícitamente que el formato plano es **lossy/best-effort** para parsing automático: es para pegar en un chat, no para que una máquina lo des-serialice sin ambigüedad. (SEC-12)
  - El nombre del fichero en la cabecera también se neutraliza (saltos de línea). (SEC-12)

## 3. RGPD / Privacidad

- **El SecretScanner procesa contenido íntegramente en local.** Es un requisito, no un detalle: el escaneo ocurre en `PastureKit`, en la máquina del usuario, **sin ninguna llamada de red.** Ningún byte del contenido escaneado (ni los secretos detectados, ni el contexto) sale de la máquina como parte del escaneo. (SEC-13)
- **El único punto en que datos del usuario salen de la máquina es Ask** (envío a Anthropic/OpenRouter), que ya existe en v1.3.0, ya tiene aviso de privacidad de una sola vez (`askPrivacyNoticeAccepted`), y va por HTTPS. **F1 refuerza este flujo añadiendo un aviso de secretos antes del envío** -- es una mejora de privacidad, no una regresión.
- **RGPD art. 25 (protección desde el diseño):** F1 es precisamente eso -- un control de minimización de exposición de datos sensibles diseñado en el flujo, no parcheado después. Bien.
- **Sin telemetría, sin analítica, sin envío de patrones detectados a ningún sitio.** Si en el futuro alguien propone "mandar estadísticas anónimas de detecciones", eso es un tratamiento nuevo y vuelve a pasar por gate. En v1.4: cero.
- **UserDefaults (F2):** las rutas de fichero son datos personales en sentido amplio (revelan nombres de proyecto). Se quedan en local, en el perfil del usuario. No es un tratamiento que requiera base legal adicional en una app mono-usuario de escritorio sin cuenta. Aceptable.

## 4. Requisitos de seguridad (SEC-N)

Numerados para que **senior-dev implemente** y **QA verifique**. Cada uno es testeable.

### F1 -- SecretScanner

- **SEC-1 — Cobertura de los tres canales de salida.** El escaneo de secretos debe ejecutarse antes de entregar contenido por CUALQUIER canal que saque datos de la app: (a) clipboard, (b) export a fichero, (c) envío a la API en Ask. Como `FeedService.deliverFeed` cubre clipboard+export pero Ask va por `AskViewModel.send → AIClient.ask`, se requieren **dos puntos de enganche** (o un servicio compartido invocado por ambos). En Ask, escanear también la pregunta del usuario, no solo el contexto. *Verificable:* test que confirma que un secreto detectable dispara el aviso en los tres canales.

- **SEC-2 — Catálogo de patrones acotado y testeado, sin catastrophic backtracking.** Las familias mínimas: Anthropic (`sk-ant-`), GitHub (`ghp_`/`gho_`/`ghu_`/`ghs_`), AWS (`AKIA[0-9A-Z]{16}`), PEM (`-----BEGIN ... PRIVATE KEY-----`), OpenAI-style (`sk-` genérico). Cada patrón debe ser anclado/lineal, sin cuantificadores anidados que permitan backtracking exponencial. *Verificable:* set de tests con un positivo conocido por familia (0 falsos negativos sobre el catálogo) y un set de negativos (prosa normal no dispara).

- **SEC-3 — Defensa ReDoS.** El escaneo debe: (a) ejecutarse fuera del main actor (no congelar la UI), (b) imponer un límite de tamaño de contenido escaneado o trabajar por líneas/streaming, y (c) usar patrones que no degraden cuadrática/exponencialmente con la longitud de entrada. *Verificable:* test que escanea un fichero de varios MB y/o una cadena patológica (p.ej. miles de `A` o de `-` sin cierre PEM) y completa en tiempo acotado (presupuesto p.ej. < 1 s), no cuelga.

- **SEC-4 — El valor del secreto NUNCA se persiste, loguea ni muestra.** El resultado de una detección expone EXCLUSIVAMENTE: tipo de patrón (etiqueta legible, p.ej. "clave Anthropic"), nombre del fichero, y localización (línea u offset). El valor coincidente no aparece en el diálogo de aviso, ni en logs, ni en mensajes de error, ni en el toast. *Verificable:* test que inspecciona el modelo de resultado del escáner y confirma que no contiene la subcadena del secreto; revisión de que ningún `print`/log incluye el match.

- **SEC-5 — Comunicar la naturaleza best-effort.** La UI del aviso (y/o su ausencia) no debe sugerir garantía. Texto del estilo "no se detectaron secretos **conocidos**", no "es seguro enviar". El escáner es una red de seguridad, no un certificado. *Verificable:* revisión del copy de la UI; QA confirma que ningún texto promete cobertura total.

- **SEC-6 — Override efímero, default seguro.** Ante detección: el diálogo NO entrega nada hasta que el usuario decide; la acción por defecto (Enter/Escape/cierre) es **Cancelar** (la segura); "Enviar de todas formas" continúa con el contenido original sin redactar. El override aplica solo a ese feed concreto y **no se persiste** (no existe un "no volver a avisar" fácil que anule la protección para siempre). *Verificable:* tests de los criterios HU-3 (cancelar/continuar/Escape).

### F2 -- Presets

- **SEC-7 — UserDefaults solo guarda rutas relativas + nombre, nunca contenido ni secretos.** El modelo de preset persiste exclusivamente: nombre del preset y lista de rutas relativas a `~/.pasture/`. Prohibido persistir contenido de fichero, API keys, o cualquier dato del Keychain en el plist de Preferences. *Verificable:* inspección del `Codable` del preset y del plist tras guardar.

- **SEC-8 — Validación del nombre de preset.** Rechazar/normalizar nombres con caracteres de control (`\0`, saltos de línea, etc.) y aplicar un límite de longitud razonable. Límite de cantidad de presets para no inflar el plist. *Verificable:* tests con nombres patológicos (control chars, vacío, muy largo) que se rechazan o saneando con mensaje claro.

- **SEC-9 — Resolución de presets pasa por PathValidator.** Al aplicar un preset, cada ruta relativa se resuelve contra `~/.pasture/` y se valida con `PathValidator.isInside(target:base:)` ANTES de seleccionar/abrir el fichero. Una ruta que escape (`../`) se descarta silenciosamente (cuenta como "fichero no disponible" del flujo HU-5), no se selecciona. *Verificable:* test que aplica un preset con una ruta `../fuera.md` y confirma que no se selecciona nada fuera de `~/.pasture/`.

### F4 -- Formatos de salida

- **SEC-10 — XML byte-idéntico a v1.3.0 (no regresión).** Tras refactorizar `ContextBuilder` para multi-formato, el formato XML produce salida byte-a-byte idéntica a v1.3.0, incluyendo el escape de `]]>` → `]]]]><![CDATA[>` y el `xmlEscapedAttribute` del nombre. *Verificable:* test de snapshot que compara la salida XML (1 fichero y N ficheros, con contenido que incluye `]]>`) contra el golden de v1.3.0.

- **SEC-11 — Markdown: fence dinámico + nombre neutralizado.** El bloque fenced envolvente usa un fence de longitud estrictamente mayor que el run de backticks más largo presente en el contenido (CommonMark). El nombre en `## name` neutraliza saltos de línea y no permite inyectar cabeceras. *Verificable:* tests con contenido que contiene ` ``` `, ` ```` `, y nombres con `\n`/`#`; el contenido del fichero nunca rompe el bloque envolvente.

- **SEC-12 — Plano: separador con nombre + documentar lossy.** El formato plano usa un separador que incluye el nombre del fichero como marca contextual, con el nombre neutralizado (sin saltos de línea). Documentar (en código y en el copy de Settings si procede) que el formato plano es best-effort para parsing automático. *Verificable:* test que confirma que el separador incluye el nombre y que un contenido que contiene la cadena separadora no produce nombres de fichero falsos en la cabecera.

### Transversal

- **SEC-13 — Cero red en F1/F2/F3/F4 salvo el envío de Ask ya existente.** El escaneo de secretos, los presets, el cálculo de límite y el formateo ocurren 100% en local. Ninguna feature de v1.4 abre una conexión de red nueva. El único egress sigue siendo `AIClient.ask` (HTTPS, ATS) en Ask, sin cambios de transporte. *Verificable:* revisión de código (ninguna `URLSession`/socket nuevo fuera de `AIClient`); test/inspección de que `SecretScanner` no importa nada de red.

### Estado por requisito

#### F1 — SecretScanner

- **SEC-1 — Cobertura de los tres canales.** ✅ Los tres canales pasan por `FeedService.guardSecrets`, que escanea off-main y solo entrega si el resultado está vacío.
  - Clipboard + export: `FeedService.executeFeed` (`FeedService.swift:36-42`) y `confirmTemplateFeed` (`:53-64`) — el escaneo precede a `deliverFeed`. ADR-002 verificado: el contenido escaneado es el **renderizado** post-templates (`:51-53`), no el crudo con `{{VARS}}`.
  - Ask: `AskView.performSend` (`AskView.swift:379-387`) escanea contexto **y** la pregunta del usuario (`SecretScanner.Input(fileName: "question", content: viewModel.question)`) antes de `viewModel.send`. El embudo es la misma `FeedService` compartida → un único diálogo en toda la app.
  - Evidencia de test: el flujo es lógica de UI; los componentes puros que lo sostienen (`scanInputs`, `guardSecrets`) están cubiertos por `SecretScannerTests` y el wiring por inspección directa del diff.

- **SEC-2 — Catálogo acotado y testeado, sin backtracking.** ✅ 6 familias en `SecretScanner.swift:124-138`, todas con prefijo literal anclado + clase de caracteres lineal con cuantificador simple `{n,}` (sin anidamiento `(a+)+`). El genérico `sk-` que en realidad es `sk-ant-` se descarta por orden + filtro explícito (`:171-173`). Patrones lineales sin backtracking catastrófico. Tests `redosLargeInput` (1 MB de cadenas patológicas: 500k `-`, 200k `A` tras `BEGIN`, 300k `a` tras `sk-`) **completa en 0.115 s**; `largeCleanInput` (4,4 MB) en 0.173 s; `sizeCapApplied` confirma el truncado. ReDoS demostrado mitigado, no asumido.
