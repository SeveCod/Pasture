## 1. Alcance y modelo de confianza

### Qué es esta superficie

Un **cuarto target SPM executable** (`pasture-mcp`), Swift puro, cero dependencias, que
habla **JSON-RPC sobre stdio** y expone `~/.pasture/` **en SOLO LECTURA** a un cliente MCP
(Claude Code / Claude Desktop). Cinco tools: `list_files`, `read_file`, `search`,
`feed_context`. Reutiliza `PastureKit` (`FileLibrary`, `PathValidator`, `ContextBuilder`,
`MDFile`). Sin escritura, sin red, sin puerto, raíz única.

### Modelo de confianza (refrendo del spike §5)

El transporte es **stdio heredado del proceso padre**. No hay socket, ni puerto, ni
conexión entrante. **Quien puede lanzar el binario ya tiene acceso al FS del usuario con
sus mismos permisos.** Esto define a quién confiamos y a quién no:

| Actor | Confianza | Nota |
|-------|-----------|------|
| El usuario (que registra el server) | Plena | El registro es el acto de consentimiento. |
| El cliente MCP (Claude Code/Desktop) que lanza el proceso | Plena en el canal, **semi-confiado en el contenido de los argumentos** | Corre con los permisos del usuario, pero **el LLM que opera el cliente alucina y puede emitir argumentos hostiles o erróneos** (rutas con `..`, queries patológicas). |
| El LLM que decide las llamadas a tools | **NO confiado** | Es la fuente de input no validado: rutas, nombres de colección, queries. Trátalo como entrada de usuario hostil. |
| Procesos de terceros en la máquina | Fuera del modelo | Si otro proceso puede lanzar nuestro binario, ya tiene acceso al FS por sí mismo. stdio no añade superficie ahí. |

**Conclusión del modelo de confianza:** la frontera de seguridad NO está entre el cliente
y el server (ambos corren como el usuario), sino entre **los argumentos que el LLM mete en
las tools** y **el filesystem fuera del vault**. Todo el threat model gira en torno a esa
frontera. Confianza cero sobre los argumentos de las tools.

### Autenticación: no, y es correcto

Refrendo el spike y el PRD (fuera de alcance §8): añadir auth a stdio sería teatro de
seguridad. No hay canal entrante que autenticar. El control es el consentimiento en el
registro + la validación de cada argumento. **Aprobado sin condiciones.**

## 2. Análisis STRIDE

Proporcional: marco cada categoría como **Aplica / No aplica** con su justificación, y
detallo solo los vectores reales.

| STRIDE | ¿Aplica? | Resumen |
|--------|----------|---------|
| **S**poofing | No (material) | stdio heredado; no hay identidad de red que suplantar. El padre es quien lanza. |
| **T**ampering | Sí | Path traversal y symlink escape (escritura NO, pero lectura fuera del vault = tampering del alcance de confianza). Output injection hacia el LLM. |
| **R**epudiation | No (material) | Solo lectura; sin estado mutable; logging mínimo a stderr. No hay acción que repudiar. |
| **I**nformation Disclosure | **Sí (núcleo)** | Exfiltración de contenido fuera del vault (traversal/symlink) y de contenido sensible **dentro** del vault (secretos -- D4). Fugas por stderr. |
| **D**enial of Service | Sí | Línea de stdin sin límite (memoria), respuesta gigante, `search` que carga N ficheros en RAM. |
| **E**levation of Privilege | No | El proceso corre como el usuario; no hay frontera de privilegio que cruzar. Solo lectura, sin exec, sin shell-out. |

A continuación, los vectores que **Aplican**, cada uno como hallazgo.

## 3. Hallazgos (vectores de ataque)

### V1 -- Path traversal en `read_file` / `feed_context` (lista de ficheros)

- **Ubicación:** tool `read_file` (arg `path`), tool `feed_context` (arg lista de ficheros). Componente: dispatcher de tools del target `pasture-mcp`.
- **Severidad:** ALTA (confianza: 95)
- **Categoría:** OWASP A01 (Broken Access Control) / A03 (Injection de ruta) / STRIDE Tampering+Info Disclosure
- **Hallazgo:** el LLM cliente puede alucinar o ser inducido a pedir `../.zshrc`, `../../../../etc/passwd`, `~/.aws/credentials`. Si la ruta se resuelve contra `~/.pasture/` y se lee sin validar, hay lectura arbitraria de ficheros del usuario.
- **Vector de ataque:** `tools/call read_file {path: "../../../.ssh/id_rsa"}`. Un prompt injection en el contenido de un fichero del vault ya leído podría inducir al LLM a pedir rutas fuera.
- **Impacto:** exfiltración de cualquier fichero legible por el usuario hacia el LLM (que hace egress a su backend). Brecha de confidencialidad grave.
- **Solución:** **toda** ruta pasa por `PathValidator.isInside(target:base:)` con `base = ~/.pasture/` **antes de cualquier I/O**. El PoC ya lo validó. **Matiz crítico verificado en el código:** `PathValidator` usa `standardizedFileURL.path`, que resuelve `..` pero **NO resuelve symlinks** -- ver V2. La ruta a validar debe construirse con `base.appendingPathComponent(arg)` y validarse la URL resultante; rechazar rutas absolutas en el argumento. Requisito: **SEC-M1**.

### V2 -- Symlink escape del vault

- **Ubicación:** `read_file`, `feed_context`, `list_files`. Componente: resolución de rutas en el dispatcher.
- **Severidad:** ALTA (confianza: 90)
- **Categoría:** OWASP A01 / STRIDE Tampering+Info Disclosure
- **Hallazgo:** `PathValidator.isInside` compara `standardizedFileURL.path`, que **no sigue symlinks**. Un symlink dentro de `~/.pasture/` que apunte a `/etc` o `~/.ssh` pasaría la comprobación `isInside` (el path del enlace está dentro del vault) pero la lectura seguiría el enlace hacia fuera. El usuario podría haberlo creado sin malicia (o un atacante con acceso de escritura al vault).
- **Vector de ataque:** `ln -s ~/.ssh/id_rsa ~/.pasture/inocente.md`; luego `read_file {path: "inocente.md"}` lee la clave privada.
- **Impacto:** misma exfiltración que V1, evadiendo la validación de path por la vía del enlace.
- **Solución:** **dos capas, en profundidad** (no confíes en una sola):
  1. **Enumeración:** `list_files` y `search` solo ven ficheros vía `FileLibrary.mdFiles`/`realSubdirectories`, que **ya filtran symlinks** (`isSymbolicLinkKey`). Verificado en el código. Mantener ese filtro como única vía de enumeración.
  2. **Lectura directa:** `read_file`/`feed_context` reciben rutas arbitrarias del LLM, NO pasan por la enumeración. Ahí hay que: (a) rechazar si el componente final o cualquier intermedio es symlink, **o** (b) validar la ruta **tras** `resolvingSymlinksInPath()` -- es decir, comprobar `PathValidator.isInside(target: url.resolvingSymlinksInPath(), base: pastureRoot.resolvingSymlinksInPath())`. La opción (b) es la robusta: resuelve el enlace y comprueba que el destino real sigue dentro del vault. Requisito: **SEC-M2**.
- **Nota:** este es el hallazgo que el spike menciona de pasada ("filtrado de symlinks") pero que NO estaba cubierto para el camino de `read_file` con ruta arbitraria. El PoC validó `..`, no symlinks en `read_file`. Lo elevo a requisito explícito.

### V3 -- DoS por línea de stdin sin límite (agotamiento de memoria)

- **Ubicación:** loop de lectura stdin del target `pasture-mcp`.
- **Severidad:** MEDIA (confianza: 85)
- **Categoría:** OWASP A04 (Insecure Design) / STRIDE DoS
- **Hallazgo:** el framing MCP es "una línea = un mensaje", delimitado por `\n`, sin `Content-Length`. Un lector ingenuo (`readLine()` acumulando hasta `\n`) ante una línea de stdin de gigabytes (cliente comprometido o bug) acumula todo en memoria antes de parsear. OOM del proceso.
- **Vector de ataque:** el cliente (o algo que escriba en el stdin del proceso) envía 2 GB sin `\n`.
- **Impacto:** caída del proceso server. Impacto acotado: es un subproceso por sesión; muere y el cliente lo relanza. No afecta a la app ni al vault (solo lectura). Por eso MEDIA, no alta.
- **Solución:** límite duro de tamaño de línea de entrada. Si una línea supera el límite sin `\n`, descartar y responder error de protocolo JSON-RPC (`-32700` parse error) o cerrar limpiamente. **Límite: 10 MB por línea/mensaje de entrada.** Justificación: un `tools/call` legítimo es de KB; 10 MB es holgura de 1000× sobre cualquier petición real. Requisito: **SEC-M3**.

### V4 -- DoS por amplificación en `search` (N ficheros en RAM)

- **Ubicación:** tool `search`, que reutiliza `MDFile.matches`.
- **Severidad:** MEDIA (confianza: 85)
- **Categoría:** OWASP A04 / STRIDE DoS
- **Hallazgo:** verificado en el código: `MDFile(url:)` **carga el contenido completo del fichero en memoria** (`String(contentsOf:)`), y `matches(query:)` hace `content.localizedCaseInsensitiveContains`. Un `search` sobre un vault grande materializa **todos** los contenidos en RAM a la vez. Con ficheros grandes o muchos ficheros, pico de memoria proporcional al tamaño total del vault.
- **Vector de ataque:** no requiere malicia: un vault grande + un `search` repetido por el LLM en bucle de refinamiento amplifica el coste.
- **Impacto:** pico de memoria / latencia. Acotado (solo lectura, subproceso). MEDIA.
- **Mitigación de diseño que SÍ está bien (refrendo D1):** la búsqueda es **literal** (`localizedCaseInsensitiveContains`), **no regex** -> **no hay ReDoS**. Refrendo D1 explícitamente: la decisión de búsqueda literal elimina por diseño el vector ReDoS del spike (R5). **Aprobado.** Una query patológica tipo `(a+)+$` se trata como texto literal -- verificado: `matches` no compila regex.
- **Solución (acotación restante):** (a) cap de tamaño por fichero leído en el escaneo de búsqueda (no cargar ficheros enormes completos: leer hasta N bytes, en línea con el cap de 2 MB del `SecretScanner`); (b) cap del número de resultados devueltos; (c) cap de longitud de la query. **Límites: 2 MB leídos por fichero en search; máx. 100 ficheros en el resultado; query máx. 1.000 caracteres.** Requisito: **SEC-M4**.

### V5 -- Respuesta gigante hacia el cliente (DoS de salida / memoria del cliente)

- **Ubicación:** `feed_context` (colección entera), `read_file` (fichero enorme), `list_files`.
- **Severidad:** MEDIA (confianza: 80)
- **Categoría:** OWASP A04 / STRIDE DoS
- **Hallazgo:** `feed_context` sobre una colección con muchos ficheros grandes ensambla un único string gigante (XML+CDATA) y lo serializa como una sola línea JSON-RPC. Sin límite, una colección de cientos de MB produce una respuesta de cientos de MB en memoria del server y del cliente. `list_files` está acotado por diseño (D5: metadatos, no contenido) -- ahí el riesgo es bajo.
- **Vector de ataque:** `feed_context {collection: <colección enorme>}`.
- **Impacto:** pico de memoria en server y cliente; posible exceso del límite práctico de mensaje MCP. Acotado. MEDIA.
- **Solución:** límite de tamaño de respuesta ensamblada. Si `feed_context`/`read_file` excede el límite, devolver `isError: true` con mensaje claro ("contexto demasiado grande, refina la selección") en vez de intentar serializar. **Límite: 25 MB por respuesta de tool.** Justificación: holgura amplia sobre un feed normal (KB-pocos MB) y por debajo de límites prácticos del transporte. Requisito: **SEC-M5**.

### V6 -- Output injection hacia el LLM (envoltura del contexto)

- **Ubicación:** `feed_context`, `read_file`. Pieza: `ContextBuilder`.
- **Severidad:** MEDIA (confianza: 85)
- **Categoría:** OWASP A03 (Injection) / STRIDE Tampering
- **Hallazgo:** el contenido del vault se entrega a un LLM. Un fichero malicioso (o un nombre de fichero malicioso) podría intentar romper la envoltura para inyectar instrucciones fuera de la marca de "esto es contexto, no instrucciones". El riesgo de prompt injection del *contenido* es inherente a entregar texto a un LLM y NO se resuelve aquí (el LLM lo procesa). Lo que SÍ se controla es que el contenido no **rompa la envoltura estructural**.
- **Vector de ataque:** fichero que contiene la secuencia de cierre CDATA `]]>`, o un nombre de fichero con saltos de línea / comillas que rompa el atributo `name`.
- **Impacto:** confusión del contexto entregado al modelo; en el peor caso, contenido que se presenta como si estuviera fuera del envoltorio de contexto.
- **Solución -- YA RESUELTA por reutilización (verificado en el código):**
  - XML: `ContextBuilder.contextTag` escapa `]]>` -> `]]]]><![CDATA[>` y el nombre vía `xmlEscapedAttribute`. Verificado.
  - Markdown: fence dinámico (backticks = run más largo + 1) y `neutralizedName` (sin saltos de línea). Verificado.
  - Plain text: `neutralizedName`. Best-effort por diseño (formato lossy).
  - **Requisito:** el target MCP **DEBE** ensamblar exclusivamente vía `ContextBuilder` (la misma pieza que la app), no reimplementar el envoltorio. Test golden que compare salida MCP con salida de la app. Requisito: **SEC-M6**. El criterio de aceptación HU-6 del PRD (escape de `]]>`) ya lo recoge.

### V7 -- Fugas por stderr (logging)

- **Ubicación:** logging del target `pasture-mcp` a stderr.
- **Severidad:** BAJA (confianza: 90)
- **Categoría:** OWASP A09 (Security Logging Failures) / STRIDE Information Disclosure
- **Hallazgo:** dos riesgos en uno. (a) **Contaminación de stdout:** un `print()` perdido a stdout rompe el protocolo (stdout es sagrado) -- es de protocolo, pero también de seguridad porque un error de framing puede filtrar fragmentos de un mensaje en otro. (b) **stderr con datos sensibles:** si el logging a stderr vuelca contenido de ficheros, rutas completas o, peor, valores que el `SecretScanner` detectaría, esos datos quedan en los logs del cliente MCP (Claude captura stderr). Eso viola el invariante v1.4 de "no-log de secretos".
- **Vector de ataque:** no es un ataque externo; es una fuga pasiva por diseño de logging descuidado.
- **Impacto:** secretos o contenido del vault en logs del cliente, fuera del control del usuario.
- **Solución:**
  - stdout: **solo** mensajes MCP válidos vía `JSONEncoder`. Todo log a stderr. (Invariante de implementación, gotcha 1-2 del spike.)
  - stderr: logging mínimo, **nunca** contenido de ficheros ni valores de argumentos crudos; rutas relativas, no absolutas; **jamás** un secreto. Reutilizar la disciplina del `SecretScanner` (que ya enmascara). Requisito: **SEC-M7**.

### V8 -- Exfiltración de secretos DENTRO del vault (la decisión D4)

- **Ubicación:** `read_file`, `feed_context`, `search`. Componente: política de contenido del canal MCP.
- **Severidad:** ALTA si el vault contiene secretos / informativa si no (confianza: 85 sobre el riesgo; la decisión va en §4)
- **Categoría:** OWASP A02 (Cryptographic Failures / exposición de credenciales) / RGPD art. 32 / STRIDE Information Disclosure
- **Hallazgo:** si el usuario guarda credenciales en `~/.pasture/` (un `.env`, una clave en una nota), un cliente MCP las leería y harían egress al backend del LLM. El `SecretScanner` de v1.4 protege los canales interactivos (clipboard/export/Ask) con "avisa y el usuario decide"; ese patrón **no existe** en un canal sin UI.
- **Vector de ataque:** el LLM pide `feed_context` de una colección que, sin que el usuario lo recuerde, contiene un fichero con una API key.
- **Impacto:** egress de credenciales hacia un tercero (el backend del LLM).
- **Solución / decisión:** **ver §4.** Es LA decisión crítica de este threat model.

## 4. LA DECISIÓN CRÍTICA -- D4: SecretScanner en el canal MCP

El PRD propone que el `SecretScanner` **NO actúe** en el canal MCP, sustituido por
consentimiento-en-el-registro + documentación. El usuario delegó en mí la validación.
La razono entera, evaluando honestamente las cuatro alternativas.

### Las alternativas, sin adornos

| Alternativa | Qué hace | Veredicto del Paranoico |
|-------------|----------|--------------------------|
| **A) Bloquear silenciosamente** | No devolver ficheros que casen un patrón de secreto. | **Rechazada.** Ficheros que desaparecen del feed sin explicación. El LLM recibe un contexto incompleto y NO sabe que falta algo; el usuario tampoco. Rompe la utilidad y, peor, lo hace de forma **invisible**: el peor tipo de fallo. Un falso positivo (`sk-` en un ejemplo de docs) elimina contexto legítimo sin rastro. |
| **B) Redactar silenciosamente** (`***`) | Sustituir el secreto por una máscara en el contenido devuelto. | **Rechazada.** Corrupción silenciosa de contexto: cambia lo que el usuario cree tener en el vault. Es **exactamente el antipatrón que v1.4 rechazó** ("no modificar el contexto que el usuario cree alimentar"). Un falso positivo corrompe texto útil sin que nadie intervenga. Coherencia con v1.4: NO. |
| **C) No escanear + documentar** (propuesta del PRD) | El canal no mira el contenido; se documenta que el vault no es para secretos; el consentimiento vive en el registro. | **Correcta en su núcleo, pero insuficiente sola.** El consentimiento-en-el-registro es un argumento válido (refrendado en §1): quien registra el server consiente exponer el vault. Pero el consentimiento es **de granularidad cero**: se da una vez, en bloque, y el usuario NO tiene visibilidad de si su vault contiene secretos *en este momento*. "Documentar y olvidar" deja al usuario a ciegas. |
| **D) Detectar y reportar SIN bloquear ni modificar** (campo `warning` no bloqueante + stat en la app) | El canal NO altera ni bloquea el contenido (entrega el feed íntegro), pero **adjunta un aviso no bloqueante** y la app muestra una estadística de "secretos detectados en el vault". | **La pieza que faltaba.** Ni corrompe (entrega íntegro), ni bloquea (no desaparece nada), ni es invisible (avisa). Es la única que respeta el invariante v1.4 *y* da visibilidad. |

### El punto técnico que cambia el análisis

Verifiqué el código de `SecretScanner`: es `nonisolated`, determinista, sin estado, con cap
de 2 MB, y **devuelve metadatos enmascarados, nunca el valor del secreto** (`SecretMatch`
lleva `maskedSnippet`, `SecretScanResult.summaryLines()` no expone valores). Es decir:
**ya existe una pieza capaz de detectar y reportar sin bloquear ni modificar.** El argumento
del PRD ("el patrón avisar-y-decidir no es trasladable a stdio") es cierto para el patrón
*interactivo y bloqueante*, pero **NO implica que haya que renunciar a toda señal**. Avisar
sin bloquear sí es trasladable: es un campo más en la respuesta, que el modelo puede leer.

¿Lo respetaría un LLM? No es el punto. El `warning` no es un control de seguridad que
dependa de que el LLM "obedezca": es **información para el usuario y trazabilidad**. Aunque
el LLM ignore el `warning`, su valor es: (1) el usuario, al ver la conversación o el log,
sabe que su vault tiene un secreto; (2) la app, con la stat visible, le avisa antes de que
sea tarde. El control de seguridad real sigue siendo solo-lectura + PathValidator; el
`warning` es **defensa en profundidad informativa**, no la única capa.

### Veredicto sobre D4: APROBADA CON CONDICIONES

**Apruebo D4 en su decisión de fondo** (NO bloquear, NO redactar, NO trasladar el patrón
interactivo al canal MCP) **pero la endurezco con dos condiciones no negociables**, porque
"no escanear + solo documentar" deja al usuario ciego ante un riesgo ALTO (egress de
credenciales a un tercero):

- **COND-D4-1 (canal):** `feed_context` y `read_file` ejecutan `SecretScanner.scan` sobre
  el contenido que van a devolver y, si hay match, **adjuntan un campo `warning` no
  bloqueante** en el `result` (NO `isError`, NO se altera ni se omite el contenido). El
  `warning` usa `summaryLines()` -- familia + fichero, **nunca el valor** (respeta SEC-4 de
  v1.4). El contenido se entrega íntegro. Esto NO es el patrón bloqueante de v1.4: es
  señalización pasiva.
- **COND-D4-2 (app):** la app muestra una **estadística visible de "secretos detectados en
  el vault"** (reusa el `SecretScanner` sobre `~/.pasture/`), de modo que el consentimiento
  del registro deje de ser ciego: el usuario que activa el server VE si su vault contiene
  patrones de secreto. Esto cierra el agujero de granularidad del consentimiento.

**Por qué con condiciones y no aprobada a secas:** la propuesta original (C) es coherente y
no introduce el antipatrón, pero su mitigación ("documentar") es la más débil de la tabla de
anti-racionalización: *"ya lo documentaremos / el usuario ya debería saberlo"* no es un
control, es una nota. Existiendo ya una pieza (`SecretScanner`) que detecta y reporta sin
corromper ni bloquear, **renunciar a ella es renunciar a defensa en profundidad gratuita**.
El coste de COND-D4-1/2 es bajo (la pieza existe, es `nonisolated`, cabe en el ensamblado) y
el beneficio es convertir un riesgo ALTO ciego en un riesgo ALTO **visible y trazable**.

**Por qué NO la bloqueo:** porque las alternativas de bloqueo (A) y redacción (B) son
**peores que el problema** -- el PRD tiene razón en eso, y lo refrendo. Bloquear el canal o
corromper el contexto haría más daño (invisible) que el riesgo que mitiga. La vía correcta
es la señal no bloqueante, no la censura.

## 5. Requisitos de seguridad (verificables por QA)

Numerados, con límites concretos. Cada uno es una aserción que QA puede convertir en test.

| ID | Requisito | Límite / criterio verificable |
|----|-----------|-------------------------------|
| **SEC-M1** | Toda ruta de `read_file` y de la lista de `feed_context` pasa por `PathValidator.isInside(base: ~/.pasture/)` **antes de cualquier I/O**. Rutas absolutas en el argumento se rechazan. | `read_file {path:"../../etc/passwd"}` y `{path:"/etc/passwd"}` -> `isError:true`, 0 lecturas fuera del vault. 100% de traversals rechazados. |
| **SEC-M2** | La validación de ruta se hace **tras resolver symlinks**: `isInside(target: url.resolvingSymlinksInPath(), base: root.resolvingSymlinksInPath())`. La enumeración (`list_files`/`search`) usa solo `FileLibrary` (que ya filtra symlinks). | Un symlink dentro del vault que apunte fuera (`ln -s ~/.ssh/id_rsa ~/.pasture/x.md`) -> `read_file {path:"x.md"}` devuelve `isError:true`, NO el contenido del destino. `list_files` no lo enumera. |
| **SEC-M3** | Límite de tamaño de mensaje/línea de entrada en el loop de stdin. | Línea de stdin **> 10 MB** sin `\n` -> el server NO acumula ilimitado; descarta y responde `-32700` o cierra limpio. Sin OOM. |
| **SEC-M4** | `search` acotado: búsqueda **literal** (no regex, reutiliza `MDFile.matches`), cap de bytes leídos por fichero, cap de resultados, cap de longitud de query. | Query `(a+)+$` se trata como literal (sin ReDoS, sin timeout). Máx **2 MB leídos por fichero** en el escaneo; máx **100 ficheros** en el resultado; query **> 1.000 caracteres** -> `isError:true` o truncado documentado. |
| **SEC-M5** | Límite de tamaño de respuesta ensamblada en `feed_context` y `read_file`. | Respuesta que excedería **25 MB** -> `isError:true` ("contexto demasiado grande, refina la selección"), NO se serializa el gigante. |
| **SEC-M6** | El ensamblado de contexto usa **exclusivamente** `ContextBuilder` (sin reimplementar envoltura). ## 7. RGPD / privacidad: cadena de responsabilidad

El server expone contenido **local** a un cliente **local**. Importa quién hace el egress
real y dónde está la responsabilidad RGPD.

### Cadena de tratamiento de datos

```
[~/.pasture/ en disco del usuario]
        │  (1) lectura local, mismo usuario, sin red
        ▼
[pasture-mcp -- proceso local, solo lectura, sin egress]
        │  (2) stdout heredado -> stdin del cliente (IPC local, sin red)
        ▼
[Cliente MCP -- Claude Code/Desktop, proceso local del usuario]
        │  (3) EGRESS REAL: envía el contexto al backend del LLM (Anthropic) por red
        ▼
[Backend del LLM -- tercero, fuera del control de Pasture]
```

### Reparto de responsabilidades

| Etapa | Quién | Responsabilidad RGPD |
|-------|-------|----------------------|
| (1) Lectura del vault | `pasture-mcp` | Tratamiento local en el dispositivo del usuario, mismo permiso que la app. **No hay egress aquí.** El server no envía nada a ninguna red -- no tiene código de red (verificable: sin `URLSession` en el target MCP). Art. 25 (privacidad desde el diseño): el alcance solo-lectura + raíz única + PathValidator es minimización por diseño. |
| (2) IPC stdio | SO / cliente | Transferencia local entre procesos del mismo usuario. No sale del dispositivo. |
| (3) Egress al LLM | **El cliente MCP (Claude), NO Pasture** | El responsable del envío a un tercero es el cliente MCP que el usuario configuró. Pasture **no** controla ni inicia ese egress. La base legal y el aviso de privacidad de ese envío son del cliente MCP (igual que hoy, cuando el usuario pega manualmente en Claude). |

### Implicaciones y obligaciones

- **Pasture NO es el responsable del egress.** El server es una fuente local que el usuario
  conecta voluntariamente a su propio cliente. El acto de **registrar** el server es el
  consentimiento del usuario a exponer su vault a su cliente AI (paralelo a la "Ask privacy
  notice" de v1.4, donde la app avisa de que el contenido se envía al proveedor).
- **Datos personales en el vault:** si el usuario guarda datos personales de terceros en
  `~/.pasture/`, exponerlos vía MCP los manda al LLM. Igual que el Feed manual de hoy. La
  mitigación es la **transparencia**: la UX de registro debe avisar, en una frase, de que el
  contenido del vault quedará accesible al cliente MCP y, por tanto, al backend del LLM.
  **Esto refuerza COND-D4-2 (SEC-M9):** la stat de secretos + el aviso de privacidad en el
  registro son el "consentimiento informado", no solo "consentimiento".
- **Art. 17 (olvido) / Art. 20 (portabilidad):** sin impacto nuevo -- el server es solo
  lectura sobre ficheros que el usuario ya controla en su disco; borrar/exportar es operar
  sobre `~/.pasture/` directamente, como hoy.
- **Art. 32 (seguridad del tratamiento):** cubierto por PathValidator + symlink filtering +
  no-egress del server + SecretScanner como señal (D4). El cifrado en tránsito del egress (3)
  es responsabilidad del cliente MCP (TLS al backend), fuera de nuestro alcance.

**Requisito de transparencia (parte de la UX de registro, P0):** la pantalla de registro MCP
muestra una frase clara: *"Al registrar este servidor, el contenido de `~/.pasture/` quedará
accesible a tu cliente MCP, que podrá enviarlo a su proveedor de IA."* Sin esto, el
consentimiento no es informado.

3. **SEC-M3/M4/M5** -- los tres límites numéricos de DoS (entrada 10 MB, search 2 MB/100/1k,

4. **SEC-M10** -- cero dependencias en el nuevo target.
5. **Aviso de transparencia** en la UX de registro (§7).
